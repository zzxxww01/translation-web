# Metadata Translation Isolation Design

## Goal

Prevent structured metadata paragraphs from being overwritten by the normal longform translation pipeline, while preserving manual editability in the UI.

Affected metadata types:

- `subtitle`
- `byline`
- `date_access`
- `source`

## Problem

The current longform flow has two translation paths:

1. `SourceMetadataTranslationService` translates structured metadata first.
2. The later full-document body translation flow can still process those same paragraphs again.

That makes metadata paragraphs unstable:

- they can receive normal `pro` / `default` draft translations after metadata translation
- they can appear inconsistent with metadata-specific rules
- later automatic runs can overwrite the intended metadata result

## Desired Behavior

Structured metadata paragraphs should be handled as follows:

1. During full-document translation, metadata paragraphs are translated only by the metadata-specific flow.
2. The body translation flow skips those paragraphs entirely.
3. Existing manual editing remains allowed.
4. If a user manually edits a metadata paragraph, later automatic full-document translation must not overwrite it.

## Recommended Approach

Use metadata-type-based exclusion in the automatic body translation path.

Why:

- stable and explicit
- does not depend on whether an earlier translation record exists
- matches existing parser/domain intent, where these paragraphs are already marked as metadata

## Implementation Scope

### 1. Define structured metadata skip rule

Create one shared rule for auto-skip behavior covering:

- `subtitle`
- `byline`
- `date_access`
- `source`

The skip rule should be used only by automatic longform translation flows, not by manual editing routes.

### 2. Apply skip rule to automatic longform flows

Update both automatic longform entry paths so that structured metadata paragraphs are excluded from normal paragraph/body translation:

- full-document SSE translation path
- four-step longform translation path

The metadata translation stage should still run before body translation.

### 3. Preserve manual editing

Do not block:

- paragraph update
- confirm
- direct translate
- paragraph-level manual retranslate

This feature only prevents automatic full-document overwrites.

### 4. Keep current metadata service behavior

Do not redesign `SourceMetadataTranslationService` in this change.

Its current responsibilities remain:

- detect and translate structured metadata
- apply metadata-specific formatting preservation
- persist metadata translations

## Data Flow

1. Parser marks structured metadata via `metadata_type`.
2. Full-document translation starts.
3. Metadata translation stage runs first.
4. Body translation iterates paragraphs.
5. If paragraph is structured metadata, skip it.
6. Otherwise translate normally.
7. Manual edits remain available afterward.

## Risks

### Misclassified metadata

If a paragraph is incorrectly marked as structured metadata, it will now be skipped by automatic body translation.

Mitigation:

- rely only on existing structured metadata types
- keep manual editing available

### Divergence between metadata and body expectations

A user may expect subtitle text to be re-generated by the main article prompt.

Decision:

- do not allow automatic body prompt overwrite
- metadata paragraphs are intentionally governed by metadata logic

## Testing

Add regression coverage for:

1. structured metadata paragraphs are skipped by automatic body translation
2. metadata-specific translations remain intact after full-document translation
3. manual edits on metadata paragraphs are preserved across later automatic runs

## Non-Goals

- redesigning metadata prompts
- changing UI behavior
- preventing manual edits to metadata paragraphs
- changing metadata parsing heuristics
