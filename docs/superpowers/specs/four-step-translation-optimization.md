# 四步翻译法优化设计文档

## 1. 优化目标

**核心目标**: 提升翻译的自然度,同时保持准确性和一致性

**关键痛点**:
- 段落组织缺少中文起承转合
- 原文语气被削平成学术腔
- 过度拆分句子失去节奏和力度

**优化原则**:
- 允许适度重组(超长句拆成2句),但不要拆成3-4个碎片
- 从"规则驱动"转向"示例驱动 + 分层审查"
- 优先解决自然度问题

---

## 2. 三层优化方案

### 第一层:高优先级(直接影响自然度)

#### 优化点 1: Step 2 注入"节奏感"和"反例库"

**当前问题**:
- `section_batch_translate.txt` 只有抽象规则,缺少具体示例
- 没有明确的句子长度指导
- 缺少语气保持的具体指令

**优化方案**:
在初译时注入以下内容:

1. **正例句对**(3-5个):
   ```
   原文: The implementation of this feature requires careful consideration of edge cases.
   ❌ 差: 这个功能的实现需要仔细考虑边缘情况。
   ✅ 好: 实现这个功能时,需要仔细考虑各种边缘情况。
   
   原文: However, this approach has several limitations.
   ❌ 差: 然而,这种方法有几个限制。
   ✅ 好: 不过,这种方法也有一些局限性。
   ```

2. **反例清单**:
   - ❌ 避免: "进行...操作" → ✅ 改为: 直接用动词
   - ❌ 避免: "对...进行分析" → ✅ 改为: "分析..."
   - ❌ 避免: 把一句话拆成3-4个短句
   - ❌ 避免: 统一用"然而"/"因此" → ✅ 改为: 根据语气选择"不过"/"所以"/"于是"

3. **句子长度指导**:
   - 原文1句 → 译文1-2句(超长句可拆分)
   - 原文2-3句 → 译文2-3句(保持节奏)
   - 避免拆成3+个短句

4. **语气保持指令**:
   - 识别原文语气(正式/轻松/幽默/严肃)
   - 选择对应的中文表达方式
   - 保持段落内语气一致性

**实施位置**: `src/prompts/longform/translation/section_batch_translate.txt`

**代码改动**: `src/llm/base.py` 的 `_build_batch_translation_prompt()` 方法需要注入示例库

---

#### 优化点 2: Step 3 三级分层审查

**当前问题**:
- `section_critique.txt` 包含40+条规则,一次性审查容易遗漏
- 没有优先级区分,注意力分散

**优化方案**:
将审查分为三轮,每轮聚焦不同优先级:

**P0 审查(致命错误,5-8条)**:
- 术语不一致(与术语表冲突)
- 漏译/错译关键信息
- 逻辑关系颠倒
- 数字/代码错误

**P1 审查(严重偏离,8-12条)**:
- 语气严重不符(正式↔口语)
- 句子过度拆分(1句→3+句)
- 书面语过重("进行"/"对...进行")
- 段落逻辑不连贯

**P2 审查(优化空间,10-15条)**:
- 用词可以更自然
- 句式可以更流畅
- 连接词可以更丰富
- 细节表达可以更地道

**实施方式**:
- 每轮输出: `[P0/P1/P2] 问题列表 + 严重程度评分(1-10)`
- 只有P0/P1问题才强制要求修复
- P2问题作为可选优化建议

**实施位置**: `src/prompts/longform/review/section_critique.txt`

**代码改动**: `src/llm/base.py` 的 `_build_reflection_prompt()` 需要支持分层审查

---

#### 优化点 3: 示例库机制

**当前问题**:
- 每个章节独立翻译,缺少风格参考
- 无法保证全文风格一致性

**优化方案**:
在 Step 0 预扫描时:

1. **提取代表性段落**(2-3个):
   - 选择标准: 包含多种句式、有代表性语气、长度适中(100-200词)
   - 覆盖不同类型: 叙述性、论证性、技术性

2. **高质量翻译**:
   - 使用最强模型(如 Claude Opus)翻译这些段落
   - 人工审核(可选)
   - 作为"风格锚点"

3. **注入到后续步骤**:
   - Step 2 初译时: 展示1-2个示例段落
   - Step 3 反思时: 对比当前译文与示例风格
   - Step 4 润色时: 参考示例的表达方式

**实施位置**:
- `src/agents/four_step_translator.py` 的 `translate_section()` 方法
- 新增 `_extract_example_sections()` 和 `_translate_examples()` 方法

**代码改动**:
```python
# 在 translate_section() 开始时
if section_index == 0:  # 第一个章节
    self.example_library = self._build_example_library(sections)
    
# 在调用 LLM 时注入示例
prompt = self._build_prompt_with_examples(
    base_prompt, 
    self.example_library
)
```

### 第二层:中优先级(提升一致性)

#### 优化点 4: Step 0 术语提取增强

**当前问题**:
- 术语表只有"原文→译文"映射
- 缺少使用场景和反例

**优化方案**:
扩展术语表格式:

```json
{
  "term": "edge case",
  "translation": "边缘情况",
  "context": "技术文档中描述异常场景",
  "alternatives": ["边界情况", "特殊情况"],
  "avoid": ["边缘案例", "边缘用例"],
  "first_occurrence": "第3章第2段",
  "frequency": 12
}
```

**实施位置**: `src/prompts/longform/analysis/article_analysis.txt`

---

#### 优化点 5: Step 1 增加段落级论证结构

**当前问题**:
- `section_role_map.txt` 只标注章节角色,没有段落级分析
- 缺少关键转折点标注

**优化方案**:
在理解阶段增加:

1. **段落级角色**:
   - 引入论点 / 展开论证 / 举例说明 / 对比分析 / 总结收束

2. **关键转折标注**:
   - 转折点: "However", "On the other hand"
   - 因果关系: "Therefore", "As a result"
   - 递进关系: "Moreover", "Furthermore"

3. **段落间关系**:
   - 并列 / 递进 / 转折 / 因果 / 总分

**实施位置**: `src/prompts/longform/analysis/section_role_map.txt`

---

#### 优化点 6: Step 4 从"最小修改"改为"策略选择"

**当前问题**:
- `paragraph_revision.txt` 强调"最小修改",导致不敢重组
- 缺少自我验证机制

**优化方案**:
改为三种策略:

1. **微调策略**(P2问题):
   - 替换个别用词
   - 调整标点符号
   - 不改变句子结构

2. **重组策略**(P1问题):
   - 合并/拆分句子(1句→2句,不超过2句)
   - 调整语序
   - 改变连接方式

3. **重译策略**(P0问题):
   - 完全重新翻译
   - 保留术语和关键信息
   - 重新组织表达

**自我验证**:
- 修改后对比原文,确认信息完整
- 检查是否过度拆分(>2句)
- 评估自然度提升程度(1-10分)

**实施位置**: `src/prompts/longform/review/paragraph_revision.txt`

### 第三层:系统性改进(长期质量)

#### 优化点 7: 反馈循环

**当前问题**:
- 每个章节独立处理,不从前面的经验中学习

**优化方案**:
在翻译第N个章节时,注入前N-1个章节的反馈:

1. **成功案例**(2-3个):
   - 原文片段 + 译文 + 为什么好
   - 例: "这里用'不过'代替'然而',语气更自然"

2. **失败案例**(1-2个):
   - 原文片段 + 初译 + 问题 + 改进后
   - 例: "初译拆成3句太碎,改为2句后节奏更好"

3. **模式总结**:
   - "这篇文章倾向于使用轻松语气"
   - "技术术语保持英文更合适"

**实施位置**: `src/agents/four_step_translator.py` 的 `translate_section()` 方法

**代码改动**:
```python
# 在每个章节完成后
self.feedback_history.append({
    'section_index': section_index,
    'good_examples': [...],
    'bad_examples': [...],
    'patterns': [...]
})

# 在翻译下一个章节时注入
if self.feedback_history:
    prompt = self._inject_feedback(prompt, self.feedback_history[-3:])
```

#### 优化点 8: 质量锚点标定

**当前问题**:
- "自然度"/"准确性"等评分标准模糊
- 不同章节的评分标准不一致

**优化方案**:
明确定义质量标准:

**自然度评分**:
- 9-10分: 完全像中文母语者写的,读起来毫无翻译腔
- 7-8分: 整体自然,偶有书面语过重或表达生硬
- 5-6分: 能理解但有明显翻译腔,如"进行...操作"
- <5分: 严重翻译腔,句式别扭,难以阅读

**准确性评分**:
- 9-10分: 信息完整,术语准确,逻辑清晰
- 7-8分: 主要信息准确,个别细节可优化
- 5-6分: 有明显错译或漏译
- <5分: 严重错译,改变原文意思

**一致性评分**:
- 9-10分: 术语统一,风格一致,段落衔接自然
- 7-8分: 整体一致,个别地方有差异
- 5-6分: 术语或风格有明显不一致
- <5分: 严重不一致,像多人翻译拼凑

**实施位置**: 所有涉及评分的 prompt 文件

## 4. 风险与缓解

### 风险 1: 示例库可能引入偏见

**描述**: 如果示例段落翻译质量不高,会影响全文

**缓解措施**:
- 使用最强模型翻译示例
- 可选人工审核
- 提供"禁用示例库"选项

---

### 风险 2: 分层审查可能遗漏问题

**描述**: 三轮审查可能比一次性审查遗漏某些问题

**缓解措施**:
- P0/P1/P2规则设计时确保覆盖全面
- 每轮审查后汇总问题
- 提供"完整审查模式"作为备选

---

### 风险 3: 反馈循环可能累积错误

**描述**: 如果前面章节翻译有问题,反馈会传播错误

**缓解措施**:
- 只注入最近3个章节的反馈
- 优先注入"成功案例"
- 提供"清除反馈历史"功能

---

## 5. 成功指标

### 定量指标

1. **自然度评分**: 从平均6.5分提升到7.5-8分
2. **过度拆分率**: 从30%降低到<5%
3. **术语一致性**: 从85%提升到95%+
4. **用户满意度**: 从7分提升到8.5分

### 定性指标

1. **语气保持**: 能准确识别并保持原文语气(正式/轻松/幽默)
2. **段落衔接**: 中文起承转合自然,不生硬
3. **句子节奏**: 长短句搭配合理,不过度拆分
4. **表达地道**: 避免"进行...操作"等翻译腔

## 6. 后续优化方向

### 6.1 上下文窗口优化

- 当前: 每个章节独立翻译
- 优化: 注入前后1-2个章节的摘要,提升衔接

### 6.2 多模型协作

- 当前: 所有步骤使用同一模型
- 优化: Step 2用快速模型,Step 3/4用强模型

### 6.3 用户反馈集成

- 当前: 无用户反馈机制
- 优化: 允许用户标注好/坏案例,加入示例库

### 6.4 领域适配

- 当前: 通用翻译规则
- 优化: 根据文档类型(技术/文学/商务)调整规则
