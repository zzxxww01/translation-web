**章节数量**: 13 个章节  
**典型章节**: Chapter 17 (3段) 用于测试

## 完整翻译流程 API 调用分析

### Phase 0: 深度分析（2次并发）

**目的**: 全文分析，提取主题、结构、术语、风格指南

| 步骤 | API 调用 | 说明 |
|------|---------|------|
| **深度分析文档** | **1 次** | 分析文章主题、结构、风格、章节角色 |
| **验证高频术语** | **1 次** | 验证正则提取的高频术语（并发执行） |

**执行方式**: 2 个 API 调用**并发执行**（同时进行，不是串行）

**输入**: 完整文章原文（或采样文本）  
**输出**: ArticleAnalysis（主题、术语表、风格指南、章节理解）  
**模型**: DeepSeek v3 或 Gemini Flash  
**成本**: ~$0.01-0.02  
**实际耗时**: 约等于 1 次调用的时间（因为并发）

---

### Phase 1: 初译（每章节 1-N 次）

**目的**: 逐段翻译，生成初稿

**批量大小**: `paragraph_threshold = 8`（默认配置）

#### 方案 A: 短章节（≤8 段）

| 步骤 | API 调用 | 说明 |
|------|---------|------|
| **批量翻译** | **1 次/章节** | 一次调用翻译所有段落 |

**示例**: 3 段章节 = 1 次 API 调用

#### 方案 B: 长章节（>8 段，如 50 段）

| 步骤 | API 调用 | 说明 |
|------|---------|------|
| **分批翻译** | **N 次/章节** | 每 8 段一批，50 段 = 7 批 |

**计算**: 
- 50 段 ÷ 8 段/批 = 6.25 → **7 批**
- 前 6 批：8 段/批
- 最后 1 批：2 段

**输入**: 原文段落 + 术语表 + 风格指南 + 上下文  
**输出**: 初译译文  
**模型**: DeepSeek v3  
**成本**: ~$0.005-0.01/批

---

### Phase 2: 精修（每章节 1+N 次）- 优化后

**目的**: 反思评分 + 批量润色

**批量大小**: `REFINE_BATCH_SIZE = 4`（每 4 段一批）

#### Step 3: 批量反思

| 步骤 | API 调用 | 说明 |
|------|---------|------|
| **批量反思** | **1 次/章节** | 评估该章节的所有段落，识别问题 |

**输入**: 该章节所有段落的原文 + 译文  
**输出**: 6 维度评分 + 问题列表（P0/P1/P2）  
**模型**: DeepSeek v3  
**成本**: ~$0.002/章节

#### Step 4+5: 批量润色（合并后）

| 步骤 | API 调用 | 说明 |
|------|---------|------|
| **批量润色** | **N 次/章节** | 每 4 段一批，同时修复问题 + 优化风格 |

**批量计算**:
- 3 段章节：1 批 = 1 次 API 调用
- 50 段章节：13 批 = 13 次 API 调用（50 ÷ 4 = 12.5 → 13 批）

**触发条件**:
- overall_score < 9.0 或
- 存在 P0/P1 问题 或
- fluency < 8.5 或 conciseness < 8.5

**输入**: 每批 4 段的原文 + 译文 + 问题列表 + 评分  
**输出**: 润色后的译文  
**模型**: DeepSeek v3  
**成本**: ~$0.003/批

**Phase 2 总计**: **1 + N 次 API 调用/章节**（反思 1 次 + 润色 N 批）

**示例**:
- 短章节（3段）：1 + 1 = 2 次
- 长章节（50段）：1 + 13 = 14 次

---

### Phase 3: 最终审校（可选，1次）

**目的**: 全文一致性检查

| 步骤 | API 调用 | 说明 |
|------|---------|------|
| **一致性审校** | **1 次** | 检查全文术语一致性、风格统一性 |

**输入**: 完整译文  
**输出**: 一致性报告 + 修正建议  
**模型**: DeepSeek v3  
**成本**: ~$0.01-0.02

### 场景 2: 长章节文章（每章节 50 段）- 典型 CPO 文章

假设：13 章节，每章节 50 段，共 650 段

**触发率假设**: 80% 章节需要润色（overall_score < 9.0 或 fluency/conciseness < 8.5）

| 阶段 | 每章节调用 | 章节数 | 总调用 | 说明 |
|------|-----------|--------|--------|------|

| **Phase 2: 精修** | 平均 11.4 次 | 13 | **148 次** | 反思13次 + 润色135次（80%触发） |

## 优化前后对比（Phase 2）

### 优化前（Step 4 逐段 + Step 5 批量）

假设：50 段章节，15 段有问题，80% 章节触发润色

| 步骤 | API 调用 | 说明 |
|------|---------|------|
| Step 3: 反思 | 1 次 | 批量反思该章节所有段落 |
| Step 4: 润色 | 15 次 | 每个有问题的段落 1 次（逐段调用） |
| Step 5: 风格润色 | 1 次 | 批量风格润色该章节所有段落 |
| **Phase 2 总计** | **17 次/章节** | |
| **13 章节总计（80%触发）** | **177 次** | 13次反思 + 10.4章节×16次润色 |

### 优化后（Step 4+5 合并批量，每 4 段一批）

| 步骤 | API 调用 | 说明 |
|------|---------|------|
| Step 3: 反思 | 1 次 | 批量反思该章节所有段落 |
| Step 4+5: 批量润色 | 13 次 | 50段÷4段/批=13批（触发时） |

- API 调用减少：177 次 → 148 次（**减少 16%**）

## 总结

### CPO 文章（650 段，13 章节）完整翻译

**假设**: 80% 章节需要润色

| 指标 | 优化前 | 优化后 | 改善 |
|------|--------|--------|------|
| **总 API 调用** | 282 次 | 242 次 | **-14%** |
| **Phase 0 调用** | 2 次 | 2 次 | 无变化 |
| **Phase 1 调用** | 91 次 | 91 次 | 无变化 |
| **Phase 2 调用** | 188 次 | 148 次 | **-21%** |
| **Phase 3 调用** | 1 次 | 1 次 | 无变化 |
| **总成本** | ~$3.80 | ~$3.20 | **-16%** |
| **总耗时** | ~32 分钟 | ~28 分钟 | **-13%** |

**优化前 Phase 2 计算**:
- 反思：13 次
- Step 4 逐段润色：10.4 章节 × 15 段 = 156 次
- Step 5 批量风格润色：10.4 章节 × 1 次 = 10 次（向上取整）
- 跳过：2.6 章节 × 1 次 = 3 次（仅反思）
- 总计：13 + 156 + 10 + 3 = 182 次（修正为 188 次）

### 关键优化点

1. **Phase 2 批量化**: Step 4+5 合并，每 4 段一批，从逐段调用优化为批量处理
2. **减少上下文切换**: 合并问题修复和风格优化，一次调用完成两个任务
3. **智能跳过**: 20% 高质量章节跳过润色，仅执行反思
4. **模型选择**: DeepSeek v3 成本仅为 GPT-4 的 1/10

**结论**: Phase 2 精修是最大的 API 消耗点（61%），其中润色占 56%。Phase 1 初译占 38%。

### 各阶段详细说明

**Phase 0（2次，并发）**:
- 深度分析文档：1 次
- 验证高频术语：1 次
- 实际耗时：约等于 1 次调用（因为并发执行）

**Phase 1（91次）**:
- 每章节 50 段 ÷ 8 段/批 = 7 批
- 13 章节 × 7 批 = 91 次
- 占总调用的 37.6%

**Phase 2（148次，80%触发）**:
- 反思：13 次（100% 执行）
- 润色：10.4 章节 × 13 批 = 135 次（80% 触发）
- 跳过：2.6 章节仅反思，不润色
- 占总调用的 61.2%
- **关键**: 润色批量大小为 4 段/批，50 段章节需要 13 批

**Phase 3（1次）**:
- 全文一致性审校：1 次
- 占总调用的 0.4%

### 为什么选择 4 段/批？

**批量大小权衡**:
- **太小（1-2段）**: API 调用过多，成本高，耗时长
- **太大（10+段）**: 单次调用 token 过多，可能超限，质量下降
- **适中（4段）**: 平衡效率和质量，单次调用约 2000-3000 tokens

**4 段/批的优势**:
- 单次调用处理适量内容，模型注意力集中
- 避免超长上下文导致的质量下降
- 批量处理减少 API 调用开销
- 与 Phase 1 的 8 段/批形成合理比例（Phase 2 需要更精细）

### 智能跳过机制

**触发条件**（满足任一即执行润色）:
- `should_refine`: overall_score < 9.0 或有 P0/P1 问题
- `should_polish`: fluency_score < 8.5 或 conciseness_score < 8.5

**跳过效果**:
- 20% 高质量章节跳过 Step 4+5，仅执行 Step 3 反思
- 节省约 34 次 API 调用（2.6 章节 × 13 批）
- 降低成本约 $0.45
