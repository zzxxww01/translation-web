# 帖子翻译系统技术手册

更新时间：2026-03-13

本文档覆盖帖子（Post）翻译的核心功能模块、API 端点、相关模板以及配置要素。与长文翻译的复杂异步工作流和四步法不同，帖子翻译以**短、平、快、尖锐**为核心设计目标，采用同步模式处理文本段落翻译、译文优化和社交媒体标题生成。

---

# 一、业务链路总览

## 1.1 三大核心功能

帖子翻译系统暴露了三个主要的 API 端点，负责从原文翻译、针对性优化到最终发布前的标题生成：

| 简称 | 入口 | 说明 |
|---|---|---|
| **翻译链路** | `POST /api/api/translate/post` | 对原帖进行高质量首遍翻译。根据原文字符自动匹配相关术语（Glossary），并拼装到 `post_translation` 模板中进行一次性生成。 |
| **优化链路** | `POST /api/api/translate/post/optimize` | 根据人工指令对现有中文译文进行重译或调整。通过携带 `conversation_history` 记录历史对话上下文，并要求模型保持极高的分析师锐度。 |
| **标题生成** | `POST /api/api/generate/title` | 根据给定的正文内容，自动生成 6 种不同维度（悬念、数据、反直觉、痛点、极简、比喻）的高质量中文标题。 |

## 1.2 模型配置与超时保护

帖子翻译使用共享的全局 LLM 实例，默认通过 `utils.llm_factory.generate_with_fallback` 进行调用，自带针对请求超限或异常时的自动模型降级策略（Primary -> Backup）。

各链路设计了独立的超时环境变量，便于根据场景灵活动态配置：

| 端点 | 环境变量 | 默认超时（秒） | 模型回退策略 |
|---|---|---|---|
| **/translate/post** | `POST_TRANSLATE_TIMEOUT` (备用: `GEMINI_TIMEOUT`) | 60秒 | `generate_with_fallback` 自动降级 |
| **/translate/post/optimize** | `POST_OPTIMIZE_TIMEOUT` (备用: `GEMINI_TIMEOUT`) | 60秒 | `generate_with_fallback` 自动降级 |
| **/generate/title** | `POST_TITLE_TIMEOUT` (备用: `GEMINI_TIMEOUT`) | 30秒 | `generate_with_fallback` 自动降级 |

# 二、术语整合机制 (Glossary Injection)

为了确保专业词汇在帖子翻译中不出现畸变，帖子系统同样接入了全局术语表，但采用了轻量级的即刻匹配策略。

**调用逻辑：** 
调用 `build_glossary_context(request.content)` (`src/api/utils/glossary.py`) 。此函数扫描请求的英文原文，把匹配到的活跃术语生成一个简洁的 glossary 字符串。

**注入点：**
- **默认模板注入：** 匹配到的术语块将被作为 `dynamic_sections` 变量，与 `post_translation` 模板拼接。
- **自定义模板注入：** 如果前端在请求中携带了 `custom_prompt`，系统将查找 `{glossary}` 占位符并替换为生成的术语数据，然后替换 `{content}` 为用户文本。

# 三、Prompt 参考

帖子系统主要包含三个关键 Prompt，由于业务形态简单，除主翻译采用文件管理外，其余使用硬编码形式（Inline Prompt）随 API 一起维护。

## 3.1 帖子翻译模板 (Post Translation)

**文件路径：** `src/prompts/post_translation.txt`  
**调用点：** `/translate/post`

**输入占位符：**
- `{text}` — 需要翻译的原文。
- `{dynamic_sections}` — 动态插入的术语块。

**规则要点：**
1. **保留锐度：** 要求还原分析师的强烈语气和判断力度。
2. **保持简短与直白：** 禁止长篇大论、宣传腔、学术腔；文本不能比原文长。
3. **消除翻译腔：** 限定不能出现“此外”、“值得注意的是”等书面语套话。
4. **术语要求：** 产品/企业代号原样保留，金融及制造术语翻译成标准中文。

**Prompt 原文内容：**

```text
你是semiAnalysis的中文编辑，负责将semiAnalysis的英文帖子（推文、社交媒体短文）翻译成高质量的中文版本。

semiAnalysis的帖子特点：短、直接、有锐度，是分析师对行业动态的即时判断，语气比长篇报告更尖锐、更直白。

## 帖子翻译原则

- **保留锐度**：帖子的尖锐观点和强判断必须原样保留。"NVIDIA is printing money while everyone else is fighting for scraps" → "英伟达在闷声赚大钱，其他家都在抢残羹剩饭"，不要文雅化成"英伟达盈利能力突出，其他厂商竞争激烈"
- **保持简短**：译文不要比原文更长。帖子追求的是冲击力，不是面面俱到
- **直白有力**：可以用更直白的中文表达，但不是口水话。"This changes everything" → "这会改变整个格局"，不是"这具有深远的影响意义"
- **分析师口吻**：保持行业专家的姿态，不是记者报道的语气

## 术语处理
- 产品/技术代号保留英文：CoWoS、HBM、EUV、GAA、FinFET、chiplet
- 公司名保留英文：TSMC、ASML、NVIDIA、Samsung Foundry
- 行业术语用中文：foundry→晶圆代工、yield→良率、fab→晶圆厂、node→制程节点
- 金融术语用中文：capex→资本支出、gross margin→毛利率、revenue→营收

## 消除翻译腔
- 拆长句，去被动，减连接词，调中文语序
- 帖子里尤其不要出现"此外""值得注意的是""综上所述"等书面连接词

## 禁止
- 不要加原文没有的观点或信息
- 不要宣传腔（"突破性""开创性""令人振奋"）
- 不要学术腔（"具有重要意义""为……奠定基础"）
- 不要使用任何 Markdown 语法标记

{dynamic_sections}

## 待翻译帖子
{text}

直接输出翻译结果，不要任何解释。
```

## 3.2 译文优化模板 (Post Optimize)

**文件路径：** 内嵌于 `src/api/routers/translate_posts.py` (`optimize_post_translation` 方法)  
**调用点：** `/translate/post/optimize`

**输入占位符：**
- `{request.original_text}` — 原始英文帖子。
- `{request.current_translation}` — 当前版本的中文翻译。
- `{context}` — （可选）最近 3 轮交互的历史对话记录，以帮助模型理解修改的背景。
- `{request.instruction}` — 本次的具体修改指令。

**规则要点：**
强制模型以原文为准继续保持分析师的“锐度”，杜绝修改后语言变滑、变得失去焦点的常见毛病，同时严格禁止任何 Markdown 语法标记输出。

**Prompt 原文内容：**

```text
你是semiAnalysis的中文编辑，优化semiAnalysis帖子的翻译质量。

## 原文
{request.original_text}

## 当前译文
{request.current_translation}
{context}
## 优化要求
{request.instruction}

## 优化原则
1. 以原文为准对照校对，发现与原文不一致或错误之处必须纠正
2. 保留分析师语气：原文的观点、判断、锐度不要弱化或磨平
3. 消除翻译腔：拆长句、去被动、减连接词、调中文语序、直接用动词
4. 术语处理：产品/技术代号保留英文（CoWoS、HBM、EUV），行业术语用中文（晶圆代工、良率、制程节点），金融术语用中文（营收、毛利率、资本支出）
5. 译文不要比原文更长，保持帖子的简洁和冲击力
6. 不要口水话、不要宣传腔、不遗漏信息

严禁使用任何Markdown语法标记。直接输出优化后的完整译文，不要任何解释。
```

### 3.2.1 快捷优化指令与指令解析

帖子系统的优化功能通常由前端发送具体的 `instruction` 或者快捷预设的 `option_id`。为了提高效率并消除前端过度包装（Wrapper）导致的提示词重叠，目前的架构将所有系统指令模板收束在后端（`src/api/routers/translate_models.py` 的 `POST_OPTIMIZE_OPTIONS`）中：

| 快捷指令 | ID | 说明 |
| --- | --- | --- |
| **可读性** | `readable` | 从句法结构层面优化译文，消除翻译腔：拆长句、改主动、减连接词、调语序等。 |
| **更地道** | `idiomatic` | 从表达层面优化译文：用具体数据代替宣传词、简化结构、拒绝公式化段落和同义词循环。 |
| **专业化** | `professional` | 提升分析师水准：保留判断力度，使用标准半导体和金融术语，删除无信息量的口水话。 |
| **自定义** | `custom` | 允许用户输入自定义要求，例如：“更克制一些”或“强化第一句的因果冲突”。 |

**后端解析机制：**
当收到 `/translate/post/optimize` 请求时，后端调用 `resolve_post_optimize_instruction`。
如果传来的是 `option_id`（如 `readable`），后端会拉取对应的数百字专业 Prompt 约束。如果传来的是自定义的 `instruction`，则直接采用这部分作为约束条件。

之后，这段指令会被完整注入到如上文 **3.2 译文优化模板** 里的 `## 优化要求` 占位符中，这样便实现了单层Prompt管控，减少了前端的逻辑耦合。

## 3.3 标题生成模板 (Title Generation)

**文件路径：** 内嵌于 `src/api/routers/translate_posts.py` (`_build_title_prompt` 方法)  
**调用点：** `/generate/title`

**输入占位符：**
- `{content}` — 需要生成标题的内容正文。
- `{normalized_instruction}` — 用户针对标题的提示（若无则默认为 "在忠实内容的前提下，尽量更有吸引力、更有记忆点。"）

**输出格式（JSON）：**
系统强制模型输出以提供 6 种不同维度视角的标题选项（<= 20汉字，无编号无Markdown）。
- `suspense`: 悬念型
- `data`: 数据型
- `counter_intuitive`: 反直觉型
- `pain_point`: 痛点型
- `minimal`: 极简短平快
- `metaphor`: 比喻/拟物型

**Prompt 原文内容：**

```text
你是semiAnalysis中文编辑，负责生成高质量中文标题。

【内容】
{content}

【用户要求】
{normalized_instruction}

【固定要求】
1. 忠实原文：不能虚构事实、不能改变结论、不能夸张误导。
2. 长度约束：每个标题必须 <= 20 个汉字，建议 12-18 个字。
3. 风格约束：简洁有信息量，避免空泛口号和模板化措辞。
4. 表达要求：可适度增强可读性与吸引力，但不能偏离内容核心。
5. 禁止项：不要输出解释，不要输出Markdown，不要加编号前缀。

请返回严格 JSON，包含以下键：
{
  "suspense": "...",
  "data": "...",
  "counter_intuitive": "...",
  "pain_point": "...",
  "minimal": "...",
  "metaphor": "..."
}
```
