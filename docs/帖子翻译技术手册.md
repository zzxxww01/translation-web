# 帖子翻译系统技术手册

更新时间：2026-04-15

本文档覆盖帖子（Post）翻译的核心功能模块、API 端点、相关模板以及配置要素。与长文翻译的复杂异步工作流和四步法不同，帖子翻译以**短、平、快、尖锐**为核心设计目标，采用同步模式处理文本段落翻译、译文优化和社交媒体标题生成。

# 一、业务链路总览

## 1.1 三大核心功能

帖子翻译系统暴露了三个主要的 API 端点，负责从原文翻译、针对性优化到最终发布前的标题生成：

| 简称 | 入口 | 说明 |
|---|---|---|
| **翻译链路** | `POST /api/api/translate/post` | 对原帖进行高质量首遍翻译。根据原文字符自动匹配相关术语（Glossary），并拼装到 `post_translation` 模板中进行一次性生成。 |
| **优化链路** | `POST /api/api/translate/post/optimize` | 根据人工指令对现有中文译文进行重译或调整。通过携带 `conversation_history` 记录历史对话上下文，并要求模型保持极高的分析师锐度。 |
| **标题生成** | `POST /api/api/generate/title` | 根据给定的正文内容，自动生成 6 种不同维度（悬念、数据、反直觉、痛点、极简、比喻）的高质量中文标题。 |

## 1.2 模型配置与超时保护

帖子翻译使用统一的 LLM 配置系统，通过 `config/llm_providers.yaml` 管理模型选择和故障转移。

详细的 LLM 模块配置说明请参阅：[LLM模块系统手册](./LLM模块系统手册.md)

### 任务默认模型

帖子翻译使用的默认模型在配置文件的 `task_defaults` 部分定义：

```yaml
task_defaults:
  post: "gemini-flash"           # 帖子翻译
  title: "gemini-flash"          # 标题生成
```

### 超时配置

各链路设计了独立的超时环境变量，便于根据场景灵活动态配置：

| 端点 | 环境变量 | 默认超时（秒） | 故障转移 |
|---|---|---|---|
| **/translate/post** | `POST_TRANSLATE_TIMEOUT` | 60秒 | 三级故障转移（Key → 模型 → Provider） |
| **/translate/post/optimize** | `POST_OPTIMIZE_TIMEOUT` | 60秒 | 三级故障转移 |
| **/generate/title** | `POST_TITLE_TIMEOUT` | 30秒 | 三级故障转移 |

# 二、术语整合机制 (Glossary Injection)

为了确保专业词汇在帖子翻译中不出现畸变，帖子系统同样接入了全局术语表，但采用了轻量级的即刻匹配策略。

**调用逻辑：**
调用 `build_glossary_context(request.content)` (`src/api/utils/glossary.py`) 。此函数扫描请求的英文原文，把匹配到的活跃术语生成一个简洁的 glossary 字符串。

**注入点：**
- **翻译链路注入：** 匹配到的术语块将被作为 `dynamic_sections` 变量，注入到 `post_translation` 模板的角色设定之后、翻译原则之前。
- **优化链路注入：** 优化链路同样调用 `build_glossary_context(request.original_text)` 获取术语约束，注入到优化 Prompt 的顶部，确保优化后术语不会发生漂移。
- **自定义模板注入：** 如果前端在请求中携带了 `custom_prompt`，系统将查找 `{glossary}` 占位符并替换为生成的术语数据，然后替换 `{content}` 为用户文本。

# 三、Prompt 参考

帖子系统主要包含三个关键 Prompt，由于业务形态简单，除主翻译采用文件管理外，其余使用硬编码形式（Inline Prompt）随 API 一起维护。

## 3.1 帖子翻译模板 (Post Translation)

**文件路径：** `src/prompts/post_translation.txt`
**调用点：** `/translate/post`

**输入占位符：**
- `{text}` — 需要翻译的原文。
- `{dynamic_sections}` — 动态插入的术语块（位于工作流程之后、语言转换策略之前）。

**设计要点：**
1. **角色定位为中英文编辑**，任务是将英文推文重写为中文社交媒体帖子，而非逐字翻译。
2. **工作流程两步走**：先分析原文的语气、姿态和信息密度，再用中文重写。
3. **中英语言转换策略**：基于形合/意合理论，提供四条系统化转换规则（化形合为意合、化被动为主动、化抽象为具体、精简冗余）。
4. **翻译原则**：还原语气强度、保持简短、保持原文姿态、忠实原文信息、数据零误差。
5. **4 组正反示例**：每组以错误类型命名（文雅化、虚词堆砌、句式膨胀、视角弱化），覆盖科技和金融两个领域。
6. **回归样例集**：帖子 Prompt 调优的人工回归样例维护在 `tests/post_translation_regression_cases.json`，用于沉淀典型英文帖子、目标译文、必须命中的表达和应避免的错误表达，便于后续持续校准“修辞转写、缩写处理、传播感、锐度保留”等能力。

**Prompt 原文内容：**

```text
你是一位资深的科技与金融领域中英文编辑。你的任务是将英文推文重写为中文版本，发布到中文社交媒体。这不是逐字翻译，而是用地道的中文重新创作同一条推文——读起来就像原生的中文帖子。

## 工作流程

1. **分析原文**：判断原文的语气（尖锐/中性/克制/口语）、姿态（分析师/创始人/记者/个人）和信息密度
2. **用中文重写**：以原文的风格和意思为锚点，写出适合中文社交媒体发布的版本。必须还原原文的语气强度、判断力度和信息密度

{dynamic_sections}

## 中英语言转换策略

英文重”形合”（依赖连接词、从句等形式连接），中文重”意合”（依赖语义连贯）。翻译时必须完成以下转换：

- **化形合为意合**：拆分英文复合句，按中文逻辑（因果、先后、轻重）重组为短句
- **化被动为主动**：被动句改为主动句或无主句，避免滥用”被”字
- **化抽象为具体**：英文多抽象名词，中文多动词。”进行分析”→”分析”，”实现增长”→”增长”
- **精简冗余**：删掉不必要的连接词（”此外””值得注意的是””综上所述”）、多余的代词（”他的””它们的”）和”的”字堆砌

## 翻译原则

- **还原语气强度**：原文多尖锐，译文就多尖锐；原文多克制，译文就多克制。不要文雅化强判断，也不要给中性陈述加戏
- **保持简短**：译文不要比原文更长。短帖追求冲击力，不是面面俱到
- **保持原文姿态**：原文是什么身份在说话，译文要保持同样的姿态
- **忠实原文信息**：不添加原文没有的观点，不遗漏，不改变立场
- **数据零误差**：数字、百分比、倍数、日期、排名必须与原文完全一致，不得四舍五入或近似处理

## 示例

### 1. 文雅化——磨平了原文锐度
原文：NVIDIA is printing money while everyone else is fighting for scraps
✓ 英伟达在闷声赚大钱，其他家都在抢残羹剩饭
✗ 英伟达盈利能力突出，其他厂商竞争激烈

### 2. 虚词堆砌——比原文长一倍
原文：TSMC's 3nm yield is now above 80%. A year ago it was below 50%.
✓ 台积电3nm良率已超80%，一年前还不到50%。
✗ 台积电的3纳米制程良率目前已经达到了80%以上的水平，而在一年前这一数字还不到50%。

### 3. 句式膨胀——两句短帖撑成一段报告
原文：Revenue came in at $35.1B, up 122% YoY. The street expected $33.9B.
✓ 营收351亿美元，同比增长122%。华尔街预期339亿美元。
✗ 该公司本季度的收入达到了351亿美元的水平，与去年同期相比实现了约122%的显著增长，超出了市场分析师此前预期的339亿美元。

### 4. 视角弱化——第一人称判断变成了第三人称陈述
原文：We believe Samsung Foundry is at least 2 years behind
✓ 我们认为三星晶圆代工至少落后两年
✗ 据分析，三星代工业务在技术进度方面存在约两年的差距

## 特殊元素
- @用户名、#话题标签、$股票代码、URL 链接：原样保留不翻译
- emoji：原样保留，不增不删
- 线程编号（如 1/7）：原样保留

## 禁止
- 不要使用任何 Markdown 语法标记

## 待翻译帖子
{text}

直接输出翻译结果，不要任何解释。
```

## 3.2 译文优化模板 (Post Optimize)

**文件路径：** `src/prompts/post_optimize.txt`
**调用点：** `/translate/post/optimize`

**输入占位符：**
- `{glossary_section}` — 动态注入的术语约束块（来自 `build_glossary_context(request.original_text)`）。
- `{original_text}` — 原始英文帖子。
- `{current_translation}` — 当前版本的中文翻译。
- `{instruction}` — 本次的具体修改指令（由 `resolve_post_optimize_instruction` 解析）。

**设计要点：**
1. **整段优化**：要求模型对整段译文进行优化并去除翻译腔，而非仅修补局部。
2. **语言转换策略逐条展开**：形合→意合、被动→主动、抽象→具体、精简冗余四条规则分别列出，确保每条获得足够权重。
3. **术语约束通过动态注入**：`glossary_section` 保证优化后术语不会漂移。
4. **Prompt 解耦**：模板从代码中提取到独立文件，通过 `prompt_manager` 加载，与主翻译模板管理方式一致。

**Prompt 原文内容：**

```text
你是一位资深的科技与金融领域中英文编辑，正在优化一条中文推文。

{glossary_section}

## 原文
{original_text}

## 当前译文
{current_translation}

## 优化要求
{instruction}

## 优化原则
1. 以原文为准：发现与原文不一致或遗漏之处必须纠正
2. 还原语气强度：原文的观点、判断、锐度不要弱化或磨平
3. 化形合为意合：拆复合句为短句，按中文逻辑重组
4. 化被动为主动：避免滥用”被”字
5. 化抽象为具体：用动词替代抽象名词结构
6. 精简冗余：删不必要的连接词、多余代词和”的”字堆砌
7. 译文不要比原文更长
8. 数字、百分比、倍数、日期必须与原文完全一致

严禁使用任何Markdown语法标记。直接输出优化后的完整译文，不要任何解释。
```

### 3.2.1 快捷优化指令与指令解析

帖子系统的优化功能通常由前端发送具体的 `instruction` 或者快捷预设的 `option_id`。为了提高效率并消除前端过度包装（Wrapper）导致的提示词重叠，目前的架构将所有系统指令模板收束在后端（`src/api/routers/translate_models.py` 的 `POST_OPTIMIZE_OPTIONS`）中。

三种预设指令按**正交维度**划分，互不重叠，与底层 `## 优化原则` 无冗余：

| 快捷指令 | ID | 维度 | 说明 |
| --- | --- | --- | --- |
| **可读性** | `readable` | 认知负荷 | 一句一事、主语明确、先结论后解释、消除”的”堆砌。聚焦读者扫读体验。 |
| **更地道** | `idiomatic` | 用词表达 | 删连接词、去公式化套路、统一概念用词、口语化替换。聚焦中文原生感。 |
| **专业化** | `professional` | 信息密度 | 不加不减、删口水话、零冗余、数据清晰。聚焦内容精准度。 |
| **自定义** | `custom` | 用户自定义 | 允许用户输入自定义要求，例如：”更克制一些”或”强化第一句的因果冲突”。 |

**各指令完整 Prompt 内容：**

**`readable`（可读性）：**
```text
请从读者认知负荷角度优化译文，让信息更容易扫读和理解：
1. 一句一事：每句话只承载一个信息点，信息密集处拆成独立短句。
2. 主语明确：避免连续无主句，让读者立刻知道“谁做了什么”。
3. 先结论后解释：核心数据或结论放句首，条件和背景放后面。
4. 消除“的”堆叠：连续三个“的”必须拆句重组。
```

**`idiomatic`（更地道）：**
```text
请从用词和表达层面优化译文，让中文更自然地道：
1. 删连接词：去掉“此外”“然而”“值得注意的是”，靠语义衔接。
2. 去公式化：不要“尽管……但仍……”“展望未来”“综上所述”等套话。
3. 同一概念前后使用同一个词，避免同义词循环替换。
4. 口语化替换：把“基于”“鉴于”“旨在”改成“根据”“考虑到”“为了”。
```

**`professional`（专业化）：**
```text
请从信息密度与精准度角度优化译文：
1. 不加不减：不要添加原文没有的修饰，也不要省略原文限定条件。
2. 删口水话：去掉不提供信息的形容词（如“突破性”“开创性”“令人振奋”）和空洞总结。
3. 零冗余：没有信息量的句子直接删除。
4. 数据清晰：一句话超过 3 个数据点时拆成多句。
```

**后端解析机制：**
当收到 `/translate/post/optimize` 请求时，后端调用 `resolve_post_optimize_instruction`。
如果传来的是 `option_id`（如 `readable`），后端会拉取对应的 4 条精准约束。如果传来的是自定义的 `instruction`，则直接采用这部分作为约束条件。

之后，这段指令会被完整注入到如上文 **3.2 译文优化模板** 里的 `## 优化要求` 占位符中，这样便实现了单层Prompt管控，减少了前端的逻辑耦合。

## 3.3 标题生成模板 (Title Generation)

**文件路径：** `src/prompts/post_title.txt`
**调用点：** `/generate/title`

**输入占位符：**
- `{content}` — 需要生成标题的内容正文。
- `{instruction}` — 用户针对标题的提示（若无则默认为 "在忠实内容的前提下，尽量更有吸引力、更有记忆点。"）

**设计要点：**
1. **角色定位为中文社交媒体运营编辑**，聚焦于生成有钩子的标题，而非文学性修辞。
2. **核心理念：钩子 > 比喻**。好标题靠具体事实和信息落差吸引读者，不靠夸张修辞或空洞悬念。
3. **明确禁止标题党**：要求标题必须包含具体信息，禁止 "震惊！""你不知道的真相" 等空洞悬念。
4. **6 + 2 结构**：6 种指定钩子维度 + 2 个自由发挥位，模型自主选择最适合内容的角度。

**输出格式（JSON）：**
系统强制模型输出 8 个标题选项（<= 20汉字，无编号无Markdown）：
- `suspense`: 悬念型——用具体的未解问题激发好奇心，问题本身必须包含信息
- `data`: 数据型——用关键数字做标题的核心锚点
- `counter_intuitive`: 反直觉型——突出违反常识或预期的事实
- `pain_point`: 痛点型——直击读者最关心的利害关系
- `minimal`: 极简型——最短最直接的事实陈述
- `contrast`: 对比型——用两个事实的反差制造张力
- `free_1`, `free_2`: 自由发挥——不限类型，模型自主选择最有传播力的角度

**Prompt 原文内容：**

```text
你是一位中文社交媒体运营编辑，负责为科技与金融领域的内容生成高质量中文标题。

目标：生成有信息量、有钩子的标题。好标题靠具体事实和信息落差吸引读者，不靠夸张修辞或空洞悬念。

【内容】
{content}

【用户要求】
{instruction}

【标题类型说明】
- suspense: 悬念型——用具体的未解问题激发好奇心，问题本身必须包含信息
- data: 数据型——用关键数字做标题的核心锚点
- counter_intuitive: 反直觉型——突出违反常识或预期的事实
- pain_point: 痛点型——直击读者最关心的利害关系
- minimal: 极简型——最短最直接的事实陈述
- contrast: 对比型——用两个事实的反差制造张力
- free_1, free_2: 自由发挥——不限类型，选择你认为最适合这条内容的角度，写出最有传播力的标题

【示例】（内容：台积电3nm良率突破80%，一年前不到50%）
{
  "suspense": "台积电3nm良率翻倍，怎么做到的",
  "data": "从50%到80%，台积电3nm只用了一年",
  "counter_intuitive": "3nm越难做，台积电反而越赚钱",
  "pain_point": "台积电良率数据出来，三星压力更大了",
  "minimal": "台积电3nm良率破八成",
  "contrast": "一年前不到50%，现在台积电3nm良率破八成",
  "free_1": "台积电3nm一年逆袭，三星还在追",
  "free_2": "台积电良率80%意味着什么"
}

【要求】
1. 忠实内容：不虚构、不改变结论、不夸张误导
2. 标题必须包含具体信息，禁止空洞悬念（如"震惊！""你不知道的真相"）
3. 每个标题 <= 20 个汉字，建议 12-18 字
4. 简洁有信息量，不要空泛口号和模板化措辞
5. 8 个标题之间角度尽量不重复
6. 禁止输出解释、Markdown、编号前缀

请直接返回 JSON 对象：
{
  "suspense": "...",
  "data": "...",
  "counter_intuitive": "...",
  "pain_point": "...",
  "minimal": "...",
  "contrast": "...",
  "free_1": "...",
  "free_2": "..."
}
```
