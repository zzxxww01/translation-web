# 术语系统迁移回滚指南

## 概述

本文档描述如何安全地回滚术语系统迁移，恢复到旧的存储系统。

## 回滚场景

在以下情况下可能需要回滚：

1. 新系统出现严重 bug
2. 数据验证失败
3. 性能问题
4. 集成问题
5. 用户反馈需要更多时间测试

## 回滚前提条件

- 旧存储系统（`glossary/`）仍然存在
- 新存储系统（`glossary_new/`）已创建
- 有完整的迁移报告和验证报告

## 回滚步骤

### 1. 评估回滚必要性

在执行回滚前，评估：
- 问题的严重程度
- 是否可以通过修复解决
- 回滚的影响范围

### 2. 通知相关人员

- 通知所有使用系统的用户
- 说明回滚原因和预计时间
- 准备回滚后的沟通计划

### 3. 执行 Dry-Run

首先执行 dry-run 模式，预览回滚操作：

```bash
python scripts/rollback_migration.py
```

检查输出，确认：
- 前提条件满足
- 操作步骤正确
- 没有意外的警告或错误

### 4. 执行实际回滚

确认 dry-run 结果后，执行实际回滚：

```bash
python scripts/rollback_migration.py --execute
```

### 5. 验证回滚结果

回滚完成后，验证：

1. 检查目录结构：
```bash
# 新存储应该被删除
ls glossary_new/  # 应该不存在

# 旧存储应该存在
ls glossary/  # 应该存在
```

2. 检查数据文件：
```bash
# 全局术语表应该存在
cat glossary/global_glossary.json | head

# 项目术语表应该存在
ls projects/*/glossary.json
```

3. 运行系统测试：
```bash
# 运行术语相关测试
pytest tests/unit/services/test_glossary_storage.py -v
```

### 6. 更新配置

如果代码已经更新为使用新系统，需要：

1. 恢复使用旧的术语管理代码
2. 更新配置文件中的路径
3. 重启相关服务

### 7. 监控系统

回滚后持续监控：
- 系统功能是否正常
- 是否有错误日志
- 用户反馈

## 回滚脚本说明

### 功能

`scripts/rollback_migration.py` 提供以下功能：

1. **前提条件验证**
   - 检查新存储是否存在
   - 检查旧存储是否存在

2. **备份检查**
   - 检查是否有备份目录
   - 如果有备份，可以恢复

3. **删除新存储**
   - 删除 `glossary_new/` 目录

4. **恢复备份**（如果存在）
   - 从 `glossary_backup/` 恢复到 `glossary/`

5. **验证回滚**
   - 确认新存储已删除
   - 确认旧存储存在且完整

### 参数

- `--execute`: 执行实际回滚（默认是 dry-run）
- `--base-path`: 指定基础路径（默认是项目根目录）

### 输出

脚本会生成：
- 控制台输出：显示回滚进度和结果
- `docs/rollback-report.json`：详细的回滚报告

## 回滚后的行动

### 短期

1. 分析导致回滚的问题
2. 制定修复计划
3. 更新测试用例
4. 准备下次迁移

### 长期

1. 改进迁移流程
2. 增强测试覆盖
3. 完善文档
4. 加强监控

## 风险和注意事项

### 数据丢失风险

- **低风险**：如果只是删除新存储，旧数据不受影响
- **中风险**：如果恢复备份，确保备份是最新的
- **高风险**：如果旧存储已被修改或删除

### 业务影响

- 回滚期间系统可能需要短暂停机
- 用户可能需要重新执行某些操作
- 已使用新系统的功能将不可用

### 预防措施

1. **始终保留旧存储**：在确认新系统稳定前不要删除
2. **创建备份**：在迁移前备份旧存储
3. **分阶段迁移**：先在测试环境验证
4. **监控指标**：设置关键指标监控
5. **回滚演练**：在测试环境演练回滚流程

## 常见问题

### Q: 回滚会丢失数据吗？

A: 不会。回滚只是删除新存储，旧存储保持不变。如果有备份，会恢复到备份时的状态。

### Q: 回滚需要多长时间？

A: 通常几秒到几分钟，取决于数据量。主要是删除和复制文件的时间。

### Q: 可以部分回滚吗？

A: 不建议。要么完全使用新系统，要么完全使用旧系统，混合使用会导致数据不一致。

### Q: 回滚后可以再次迁移吗？

A: 可以。修复问题后，可以重新运行迁移脚本。

### Q: 如何确保回滚安全？

A: 
1. 先执行 dry-run
2. 在测试环境验证
3. 确保有完整备份
4. 准备好回滚后的验证步骤
