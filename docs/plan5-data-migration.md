# Plan 5: 数据迁移和系统切换

## 目标

将现有的术语数据从旧系统迁移到新的术语管理系统，并完成系统切换。

## 背景

当前项目中存在旧的术语管理代码和数据格式，需要：
1. 识别所有需要迁移的数据
2. 将数据转换为新格式
3. 验证迁移的正确性
4. 清理旧代码
5. 提供回滚方案

## 任务分解

### Task 5.1: 数据审计和迁移计划
**目标**: 识别所有需要迁移的数据和代码

**输出**:
- 旧系统数据清单
- 旧代码清单
- 迁移映射文档
- 风险评估

**验收标准**:
- 完整的数据清单文档
- 明确的迁移策略
- 识别所有潜在风险

### Task 5.2: 实现迁移脚本
**目标**: 创建自动化迁移脚本

**文件**:
- `scripts/migrate_terminology.py` - 主迁移脚本
- `scripts/migration_utils.py` - 迁移工具函数

**功能**:
1. 读取旧格式术语数据
2. 转换为新格式（Term + TermMetadata）
3. 生成确定性 UUID（使用 uuid5）
4. 保持引用关系
5. 支持 dry-run 模式
6. 生成详细的迁移报告

**验收标准**:
- 支持 dry-run 模式
- 生成详细日志
- 处理所有边界情况
- 保持数据完整性

### Task 5.3: 实现验证脚本
**目标**: 验证迁移的正确性

**文件**:
- `scripts/validate_migration.py` - 验证脚本

**验证项**:
1. 文件结构完整性
2. 术语数量一致性
3. 引用关系正确性
4. 数据格式正确性
5. 元数据完整性

**验收标准**:
- 所有验证项通过
- 生成验证报告
- 识别所有不一致

### Task 5.4: 执行迁移
**目标**: 在真实数据上执行迁移

**步骤**:
1. 备份现有数据
2. 运行 dry-run 验证
3. 执行实际迁移
4. 运行验证脚本
5. 生成迁移报告

**验收标准**:
- 所有数据成功迁移
- 验证脚本通过
- 备份可用

### Task 5.5: 清理旧代码
**目标**: 移除旧的术语管理代码

**文件**:
- `scripts/cleanup_old_system.py` - 清理脚本
- 识别并移除旧代码

**验收标准**:
- 旧代码已移除
- 所有测试仍然通过
- 文档已更新

### Task 5.6: 回滚方案
**目标**: 提供紧急回滚能力

**文件**:
- `scripts/rollback_migration.py` - 回滚脚本

**功能**:
1. 从备份恢复数据
2. 恢复旧代码（如果需要）
3. 验证回滚结果

**验收标准**:
- 回滚脚本可用
- 测试回滚流程
- 文档化回滚步骤

## 数据格式映射

### 旧格式
```json
{
  "term": "GPU",
  "translation": "图形处理器",
  "context": "...",
  "notes": "..."
}
```

### 新格式
```python
# Term
Term(
    id="uuid-generated",
    original="GPU",
    translation="图形处理器",
    strategy="TRANSLATE",
    source_lang="en",
    target_lang="zh"
)

# TermMetadata
TermMetadata(
    term_id="uuid-generated",
    scope="project",
    project_id="project-id",
    term_original="GPU",
    term_translation="图形处理器",
    created_at=datetime.now(),
    updated_at=datetime.now()
)
```

## 风险和缓解

### 风险 1: 数据丢失
**缓解**: 
- 完整备份
- dry-run 模式
- 验证脚本

### 风险 2: 引用关系破坏
**缓解**:
- 确定性 UUID 生成
- 引用关系验证
- 详细日志

### 风险 3: 迁移失败
**缓解**:
- 回滚脚本
- 分步执行
- 充分测试
