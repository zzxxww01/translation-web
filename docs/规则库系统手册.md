# 一、文档范围与边界

当前这套能力也可称为“学习链路”或“规则自学习机制”。与术语库这种结构化 JSON 管理不同，规则库保存的是**自然语言 bullet list**，承载跨项目可复用的翻译偏好。

# 二、核心概念

## 2.1 什么是规则库

术语库解决的是“某个词该怎么翻”，规则库解决的是“句式应该怎样组织、语气应该怎样拿捏、哪些表达习惯应该避免”。

规则示例：
- `避免“进行了……”句式，改用直接动词`
- `技术对比时保持原文判断力度，不要弱化为“可能”“或许”`
- `长定语从句优先拆成短句，避免中文阅读负担过重`

不属于规则库的内容：
- `"wafer" 翻译为 "晶圆"`
- `"CoWoS" 首次出现时加注释`

这类内容属于术语策略或术语表，不应进入规则库。

## 2.2 当前设计原则

- 规则库是**全局单库**，不是项目级多库。
- 规则库只保留**风格 / 句法 / 语气 / 流畅度**偏好。
- 术语翻译与术语注释规则属于术语库，不属于规则库。

# 三、物理存储结构

## 3.1 目录结构

```text
project_root/
├── data/
│   └── global_memory.md
```

## 3.2 存储说明

唯一规则文件是：
- `data/global_memory.md`

特点：
1. 三条学习链路提取出的规则全部汇总到这一份文件。
2. 文件格式是纯 Markdown bullet list，每行一条规则。
3. 追加时做精确字符串去重。

# 四、规则的提取来源

所有规则提取都以 `asyncio.create_task` 的方式异步执行，不阻塞主翻译请求。

## 4.1 人工修改译文

- 触发点：`projects_paragraphs.py`、`confirmation_service.py`
- 条件：AI 译文与人工修改结果 diff 比例 `>= 5%`
- Prompt：`correction_rule_extraction.txt`
- 输出：最多 5 条可复用规则

## 4.2 四步法 critique 学习

- 触发点：`four_step_translator.py`
- 条件：critique 有 `issues`
- Prompt：`reflection_rule_extraction.txt`
- 输出：最多 5 条可复用规则

## 4.3 重译指令学习

- 触发点：`confirmation_translation.py`
- 条件：用户提供 instruction，且重译前后 diff 比例 `>= 5%`
- Prompt：`retranslation_rule_extraction.txt`
- 输出：最多 3 条可复用规则

## 4.4 去重方式

新规则写入 `data/global_memory.md` 前，会做精确字符串匹配去重。完全重复的规则不会重复写入。

# 五、规则的消费与 Prompt 注入

`memory_service.get_rules_for_prompt()` 负责读取 `data/global_memory.md`，并在注入前做截断控制。

当前限制：
- 最多 20 条
- 最多 600 字符

这些规则随后通过 `prompt_builder._build_rules_section()` 拼入 `## 已学习的翻译规则` 区块。

---

# 六、规则库自动梳理

规则长期积累后会出现重复、冲突、模糊表述，因此系统提供后台梳理机制。

## 6.1 触发条件

每次成功写入规则后，`memory_service._maybe_consolidate()` 会判断：
- 当前规则数是否 `>= 8`
- 是否命中 `5%` 概率触发

命中后会异步调起 `_consolidate_rules()`。

## 6.2 梳理内容

Prompt：`rules_consolidation.txt`

LLM 会对整个全局规则库执行：
1. 合并重复规则
2. 解决矛盾规则
3. 删除模糊、不可执行规则
4. 删除术语型规则
5. 控制总量不超过 30 条

输出结果会直接覆盖 `data/global_memory.md`。

# 七、规则与术语的边界

当前实现明确将规则库和术语库解耦：

- 三个规则提取 Prompt 都明确禁止输出术语翻译规则。
- `rules_consolidation.txt` 也会删除已有术语型规则。
- `memory_service.py` 不再从规则文本中抽取术语候选。
- `terminology_review_service.py` 不再合并来自规则库的术语候选。

因此，当前系统不存在“规则自动长出术语候选，再回灌术语审阅”的链路。

# 八、管理 API

当前规则管理 API 已改为**全局路由**：

| 端点 | 方法 | 说明 |
|---|---|---|
| `/translation-rules` | `GET` | 查看所有全局规则 |
| `/translation-rules/{index}` | `DELETE` | 按索引删除全局规则 |
| `/translation-rules` | `POST` | 手动添加一条全局规则 |

`POST /translation-rules` 请求体：

```json
{
  "text": "避免连续使用多个被动句"
}
```
