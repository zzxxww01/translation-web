# 一、文档范围与边界

这套系统被称为“学习链路”或“规则自学习机制”。与“术语库”这种高度结构化（JSON、字段级管理）的体系不同，“规则库”是以**自然语言（Markdown 列表）**为载体的软性约束。

本文覆盖：
1. 规则提取的三大触发来源与条件（人工纠错、四步法反思、重译指令）。
2. 规则的持久化存储结构（项目级与全局）。
3. 规则的消耗：在翻译时如何被 Prompt Builder 截断并注入。
4. 规则的治理：后台自动梳理（Consolidation）机制。
5. 规则与术语库的跨界联动（提取规则中的术语候选）。
6. 相关管理 API。

# 二、核心概念

## 2.1 什么是规则库（Learned Rules）

术语库针对的是“名词要怎么翻译”，而规则库针对的是“句法应该怎样组织、语气应该怎样调整、哪些特定句式需要避免”。

系统从人工修改、LLM 自反思中自动提取出这种“经验”，存储为自然语言条目，随后变成后续段落翻译时的辅助 prompt。

**规则格式示例：**
- `"wafer" 翻译为 "晶圆"，不要译为 "硅片"`
- `避免 "进行了……" 句式，改用直接动词`
- `"CoWoS" 首次出现时加注释，后续直接使用`

## 2.2 存储形态的特殊性

术语库是 JSON 对象，规则库是**纯 Markdown bullet list**。
这种设计是因为 LLM 对于软性翻译原则（如“去掉冗余转折词”）的理解更依赖自然语言指令，而非键值对。

# 三、物理存储结构

## 3.1 目录结构

```text
project_root/
├── data/
│   ├── global_memory.md                        # 全局规则库（跨项目复用）
│   └── memory/
│       ├── <project_id>.md                     # 项目级规则库
│       └── <project_id>_term_candidates.json   # 从规则中自动提取出的术语候选（中间产物）
```

## 3.2 存储文件说明

1. **项目级规则文件**：`data/memory/<project_id>.md`
   - 每个项目独立的一份自学习规则列表。
   - 所有单项目内的段落修改、四步法自评都在这里累加。
2. **全局规则文件**：`data/global_memory.md`
   - （当前系统保留的扩展点）跨项目复用的翻译习惯池。
3. **关联产物**：`data/memory/<project_id>_term_candidates.json`
   - 这不是规则文件，而是“规则长出术语”的临时暂存载体，供术语审阅使用。

# 四、规则的提取来源（学习链路）

每次翻译结果被干预或遇到质量问题时，系统会触发后台提取规则。所有提取任务都是 **fire-and-forget 异步** 执行（`asyncio.create_task`），无论成功或失败都不会阻塞主请求链路。

## 4.1 来源一：人工修改段落（Correction）

- **触发方**：前端文档页段落编辑确认（`projects_paragraphs.py`） / 确认流段落人工修改（`confirmation_service.py`）。
- **前置条件**：人工修改后的译文与 AI 原译文 **Diff 差异率 $\ge$ 5%**。
- **Prompt**：`correction_rule_extraction.v2.txt`（使用 Flash 模型）。
- **行为**：LLM 对比原文、旧译文和人工结果，总结出人工编辑的偏好规律。输出最多 5 条规则。

## 4.2 来源二：四步法 Critique 自评（Reflection）

- **触发方**：四步法中的 Phase 3 Critique（`four_step_translator.py`）。
- **前置条件**：`overall_score < 8.0` 且发现了具体的 `issues`。
- **Prompt**：`reflection_rule_extraction.v2.txt`（使用 Flash 模型）。
- **行为**：通过给出的 issues 反思，提取下次翻译需要避免的模式。输出最多 5 条规则。

## 4.3 来源三：用户设定指令的重译（Retranslation）

- **触发方**：确认流或普通段落带有 `instruction` 进行重译（`confirmation_translation.py`）。
- **前置条件**：有重译 instruction，且重译结果差异 **Diff 差异率 $\ge$ 5%**。
- **Prompt**：`retranslation_rule_extraction.v2.txt`（使用 Flash 模型）。
- **行为**：根据 instruction 和前后的变化结果，提炼出具有普适指导意义的规则。输出最多 3 条规则。

## 4.4 去重机制

这三个来源提取出的规则文本，在写入项目 `.md` 文件前，会做**精确字符串匹配去重**。如果已存在完全一致的字符串，则跳过不写入。

# 五、规则的消耗与注入（如何影响翻译）

存下来的规则会在随后的新段落翻译中自动成为约束条件。

## 5.1 获取规则

`memory_service.get_rules_for_prompt(project_id)` 会读取 `data/memory/<project_id>.md`。

## 5.2 截断策略

为防止长年累月积攒的规则耗尽 context 窗口并过度稀释核心 prompt 注意力，应用层做了严格的截断：
- **最大条数限制**：`MAX_LEARNED_RULES_IN_PROMPT = 20`
- **最大字符限制**：`MAX_LEARNED_RULES_CHARS = 600`

若超过这两个限制，则仅取文件前面的规则。

## 5.3 Prompt Builder 注入区块

`src/prompts/prompt_builder.py` 里的 `_build_rules_section()` 会将截断后的规则转换成标准的 Markdown 区块，放在翻译 Prompt 原文上方：

```markdown
## 已学习的翻译规则
- 优先将长句控制在 30 字以内，超长句必须拆分。
- "wafer" 这里指芯片制造中的硅片基底。
```

> **注意：** 该区块仅在“当前项目确实有规则记录”时生成。

# 六、规则库自动梳理（Consolidation）

随着项目推进，不仅会触发最大 20 条上限的截断，部分通过局部差异提炼出来的规则，时间长了也会产生自相矛盾的情况。因此，规则库内置了治理机制：

## 6.1 触发机制

每次成功追加新规则后，`memory_service._maybe_consolidate()` 判断是否满足触发条件：
- **容量条件**：当前规则总数 $\ge$ 8 条。
- **概率触发**：满足容量时，并非次次全量重写，而是以 **5% 概率** 异步触发（避免频繁调用）。

## 6.2 执行操作

- **模型选用**：触发 `rules_consolidation.v2.txt`（使用 Flash 模型）。
- **清理逻辑**：LLM 审查整个项目规则库的内容，进行：
  1. 合并内容重复或高度接近的规则
  2. 解决存在矛盾的规则
  3. 删除表述模糊、无效的规则
  4. 强制删减，使得最终保留总条数 $\le$ 30。
- **结果覆盖**：直接将梳理后的精炼 bullet list 覆盖写回源 `.md` 文件。

# 七、跨界联动：规则转术语候选

并非所有提取出的经验都只是抽象的句子层级习惯。很多时候，人工修改只是改了一个词的意思。

## 7.1 正则捕获

在每次写入规则后，系统会主动调用 `memory_service._extract_term_candidates_from_rules()`。
这是一个轻量正则处理，它会在规则文本中寻找如下等特征的结构：
- `"X" 翻译为 "Y"`

把匹配到的英文原文（`original`）和中文（`translation`）提取出来。

## 7.2 后续汇入术语审阅

生成的配对被存在 `data/memory/<project_id>_term_candidates.json` 中。
- 这个文件不是立刻生效的翻译大纲（那归真正的 Glossary 管）。
- 当系统准备进行（四步法）段落的正式 prescan 或者进入终端用户的**术语审阅管理阶段（Terminology Review）**时，`terminology_review_service._merge_rule_candidates()` 会把这些 JSON 配对当成新收集的“候选术语”呈现给译员。
- 译员点击接受后，就会真正变成 `projects/<project_id>/glossary.json` 里的结构化 `GlossaryTerm`。

这就形成了 **软提示规则 $\to$ 结构化术语** 的升维降噪链路。

# 八、相关管理 API

系统提供了开放给前端或开发调试查看和编辑规则的端点，这些直接操作于 markdown 文件：

| 端点 | HTTP 方法 | 说明 |
|---|---|---|
| `/{project_id}/translation-rules` | `GET` | 读取整个项目级的翻译规则列表 |
| `/{project_id}/translation-rules/{index}` | `DELETE` | 根据前端提供的行号索引（index），删除指定位置的那一条规则 |

*（注：因为规则是纯文本，这里的修改是基于 List 中 index 的数组替换）*
