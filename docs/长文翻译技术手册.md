# 一、业务链路总览

## 1.1 三条翻译链路

| 简称 | 入口 | 说明 |
|---|---|---|
| **链路 A** | `POST /api/projects/{id}/translate-stream` | 文档页普通全篇翻译。逐段调用 `TranslationAgent`，**不做**全文分析、section role 分析、术语预扫描。上下文最少（仅术语表 + 前后文 + 学习规则）。 |
| **链路 B** | `POST /api/projects/{id}/translate-four-step` | 文档页四步法全篇翻译。先做全文分析 + section role 分析，再逐 section 执行 prescan → draft → critique → revision → style polish（可选）。上下文最完整。 |
| **链路 C** | `POST /api/projects/{id}/translate-all` | 确认流项目级翻译。先做全文分析 + section role 分析 + prescan，然后按 section 批量翻译（`section_batch_translate`）。不走段落级 prompt builder，走独立模板。 |

## 1.2 模型配置

### 设计原则

当前系统使用**统一的 YAML 配置文件**管理所有模型，支持多供应商、主备 Key 和三级故障转移。前端 UI 和 API 请求体中**没有** model 参数，所有翻译任务使用配置文件中的默认模型。

详细的 LLM 模块配置说明请参阅：[LLM模块系统手册](./LLM模块系统手册.md)

### 配置文件位置

`config/llm_providers.yaml`

### 任务默认模型

当前长文翻译使用的默认模型在配置文件的 `task_defaults` 部分定义：

```yaml
task_defaults:
  longform: "deepseek-v3"        # 长文翻译默认模型
  post: "gemini-flash"           # 帖子翻译
  title: "gemini-flash"          # 标题翻译
  metadata: "gemini-flash"       # 元数据提取
  analysis: "gemini-pro"         # 全文分析
```

### 故障转移机制

系统实现了三级故障转移策略：

1. **Level 1: Key 轮换** - 在同一个 Provider 的同一个模型上，尝试其他 API Key
2. **Level 2: 同组模型切换** - 在同一个 Provider 内，切换到其他可用模型
3. **Level 3: 跨组故障转移** - 切换到其他 Provider 的模型

详细说明请参阅：[LLM模块系统手册 - 故障转移策略](./LLM模块系统手册.md#五故障转移策略)

### 修改模型配置

如需修改长文翻译使用的模型，编辑 `config/llm_providers.yaml` 中的 `task_defaults.longform` 字段即可。所有翻译步骤（预扫描、初译、反思、润色、格式修复）都会使用相同的默认模型。

# 二、术语库系统

术语库的完整数据模型、存储结构、API 接口、标签体系、前端管理入口和调试方法请参阅：
- [术语库系统手册](./术语库系统手册.md)

以下仅记录与翻译链路直接相关的术语集成逻辑。

## 2.1 术语数据模型（速查）

| 模型 | 文件 | 用途 |
|---|---|---|
| `GlossaryTerm` | `core/models/glossary.py` | 持久化术语条目（original / translation / strategy / note / scope / source / status） |
| `Glossary` | `core/models/glossary.py` | 术语表容器，`get_term()` / `add_term()` |
| `EnhancedTerm` | `core/models/analysis.py` | LLM 分析产出的术语（全文分析阶段） |
| `PrescanTerm` | `core/models/memory.py` | Flash 预扫描发现的术语 |
| `TermConflict` | `core/models/memory.py` | 术语冲突（已有翻译 vs 新翻译，含上下文和词义说明） |
| `TerminologyVersion` | `core/models/memory.py` | 运行时术语版本管理（含冲突检测和解决） |

翻译策略（`TranslationStrategy`）：`preserve`（保留英文）/ `preserve_annotate`（保留英文，首次加中文注）/ `first_annotate`（首次加注英文原文）/ `translate`（直接翻译）

## 2.2 术语加载与 Prompt 注入

**加载合并：** `GlossaryManager.load_merged(project_id)` → 项目术语覆盖全局术语（case-insensitive）

**链路 B/C 额外合并：** `batch_translation_service._merge_analysis_with_project_glossary()` 把全文分析产出的术语与项目术语合并。合并优先级：项目手动术语 > 全文分析术语 > 全局术语。当 glossary 中存在与 analysis 同名的术语时，只覆盖 `translation`、`strategy`、`first_occurrence_note` 三个字段，保留 analysis 提供的 `context_meaning` 和 `rationale` 语义信息。

**Prompt 注入统一入口：** `src/core/glossary_prompt.py`
- `select_glossary_terms_for_text()` — 只选当前段落命中的 `active` 术语，使用 `_count_term_occurrences()` 做统一的正则边界匹配（英文/缩写术语按词边界校验），避免短词误命中
- `render_glossary_prompt_block()` — 统一渲染为 prompt 术语约束块
- `build_term_usage_from_project()` — 扫描当前段落之前的所有已译段落，构建术语使用记录（用于确认流重翻等跨会话场景）
- 上限 `MAX_GLOSSARY_TERMS_IN_PROMPT = 30`

**链路 A** 通过 `select_glossary_terms_for_text()` 直接注入；**链路 B/C** 通过 `LayeredContextManager._get_relevant_terms()` 做段落级过滤后注入。链路 B 的批量翻译路径（`_build_batch_context`）也会通过 `_apply_term_tracker_corrections()` 将 `term_tracker` 中已使用的翻译修正到术语列表上，与单段翻译路径保持一致。

**FIRST_ANNOTATE / PRESERVE_ANNOTATE 动态降级：** 通过 `term_usage` 参数，`first_annotate` 和 `preserve_annotate` 策略的术语在首次标注后自动改为纯翻译形式，避免重复标注。`term_usage` 的来源因链路而异：
- **链路 B/C（四步法 / 确认流批翻）：** 由 `TermUsageTracker.used_translations` 提供。`_translate_batch` 在每个段落翻译完成后立即调用 `record_translation()` 更新 tracker，后续批次通过 `_apply_term_tracker_corrections()` 感知已用翻译。断点续传时通过 `_prepopulate_term_tracker()` 预填充。
- **确认流重翻：** 由 `build_term_usage_from_project()` 在每次重翻前动态扫描当前段落之前的所有已译段落构建。
- **链路 A（普通翻译）：** 通过 `TranslationContext.term_usage` 传递（当调用方提供时）。

对于 section-batch（链路 C）模式，章节 prompt 也会消费 `term_usage`，并在每个章节完成后把本章已出现的 `first_annotate` / `preserve_annotate` 术语写回 tracker，因此“首次标注”按全文而不是按章节重复。

## 2.3 术语预扫描（Prescan）

**Prompt：** `section_prescan.txt`（#5），Flash 模型

**触发：** 链路 B（四步法 Phase 1）和链路 C（确认流），逐 section 调用

**流程：**
1. section 原文 + 已有术语表发给 Flash
2. Flash 返回 `new_terms` + `term_usages`
3. `context_manager.add_terms_from_prescan()` 合入运行时术语版本
4. 新术语与已有术语翻译冲突 → 生成 `TermConflict`

**长内容自动分段：** section >7000 字符时自动按段落边界分割，分别调用 Flash，合并去重。

**结果回流：** prescan 新术语同步回流到 `article_analysis.terminology`。

**并发预执行：** 链路 C 在无术语冲突回调时，翻译前用 `asyncio.gather` 并发预跑所有 section 的 prescan，缩短总耗时。

## 2.4 四步法实时术语冲突

链路 B 的 section prescan 发现术语冲突时，走一轮实时 SSE 对话：

1. 后端发送 `term_conflict` 事件
2. 前端暂停翻译，弹出冲突对话框（展示译法、词义说明、上下文）
3. 用户选择后 `POST /api/projects/{project_id}/resolve-conflict-live`
4. 后端更新运行时术语版本，继续翻译

更新的是**运行时术语版本**，不是直接写项目术语文件。持久化管理走术语中心和术语审阅。

# 三、学习链路（翻译规则自学习）

## 3.1 概述

学习链路从三种来源自动提取可复用翻译规则，存储为自然语言 markdown 条目，在后续翻译时注入 prompt。

**规则格式示例：**
```
- 避免 "进行了……" 句式，改用直接动词
- 技术对比时保持原文的强判断语气，不要弱化为 "可能" "或许"
- 长定语从句优先拆成短句，避免中文阅读负担过重
```

## 3.2 规则提取来源

| 来源 | Prompt | 触发条件 | 调用方 |
|---|---|---|---|
| 用户手动修改译文 | #11 `correction_rule_extraction` | diff ≥ 5% | `projects_paragraphs.py` / `confirmation_service.py` |
| 四步法 critique 发现问题 | #12 `reflection_rule_extraction` | critique score < 8.0 且有 issues | `four_step_translator.py` |
| 用户重译时给了指令 | #13 `retranslation_rule_extraction` | 有 instruction 且 diff ≥ 5% | `confirmation_translation.py` |

所有提取均为 **fire-and-forget 异步**（`asyncio.create_task`），不阻塞主请求。

## 3.3 规则存储

- **存储位置：** `data/global_memory.md`
- **格式：** 纯 markdown bullet list，每行一条规则
- **去重：** 追加前做精确字符串匹配，跳过已存在的规则
- **边界约束：** 规则库只保留风格 / 句法 / 语气 / 流畅度规则；术语翻译规则不再进入规则库，也不会再从规则库反向提取术语候选

## 3.4 规则注入翻译 prompt

`memory_service.get_rules_for_prompt()` → 读取 `data/global_memory.md` → 截断（上限 20 条 / 600 字符，常量 `MAX_RULES_IN_PROMPT` 和 `MAX_RULES_CHARS`） → 通过 `prompt_builder._build_rules_section()` 拼入 `## 已学习的翻译规则` 区块（见 6.3.3）。

## 3.5 自动梳理

每次成功提取规则后，以 **5% 概率**触发 `rules_consolidation`（#14）：
- **前置条件：** 规则库 ≥ 8 条
- **操作：** LLM 审查整个全局规则库，合并重复、解决矛盾、删除模糊规则、删除术语型规则、控制总量 ≤ 30
- **执行方式：** `asyncio.create_task`，异步非阻塞
- **结果：** 梳理后的规则直接替换原 `.md` 文件

## 3.6 管理 API

| 端点 | 方法 | 说明 |
|---|---|---|
| `/translation-rules` | GET | 查看所有全局规则 |
| `/translation-rules/{index}` | DELETE | 按索引删除全局规则 |
| `/translation-rules` | POST | 手动添加一条全局规则 |

# 四、Prompt 参考

## 分析阶段（Phase 0）

### 1. article_analysis.txt — 全文分析

**文件：** `src/prompts/longform/analysis/article_analysis.txt`

**语言：** 英文（输出英文 JSON） · **模型：** pro · **链路：** B / C

**调用点：** `deep_analyzer.py` → `base.py`

**输入占位符：**
- `{sections_outline}` — 文章各 section 的大纲
- `{text}` — 原文采样片段

**7 步任务：**
1. 一句话核心主题
2. 3-6 个论证要点
3. 文章结构描述
4. 关键术语提取（精选，不罗列常见词）
5. 原文风格 + 中文译文语体定调
6. 翻译风险点
7. 4-7 条具体翻译指南
8. GPU/半导体重文章的领域术语特殊处理
9. 隐喻、类比和文化典故识别（对中文读者不透明时标注适配策略）
10. 跨域术语冲突分析（如 GPU + 金融 + 云计算混合文章）
11. 修辞语气标记（讽刺、强调、戏剧效果等，帮助下游翻译匹配力度）

**输出 JSON 字段：**
| 字段 | 类型 | 说明 |
|---|---|---|
| `theme` | string | 核心主题 |
| `key_arguments` | string[] | 论证要点 |
| `structure_summary` | string | 结构描述 |
| `terminology` | object[] | 术语表（term / context_meaning / translation / strategy / first_occurrence_note / rationale） |
| `style` | object | 风格（tone / target_audience / translation_voice） |
| `challenges` | object[] | 翻译风险（type / location / issue / suggestion），type 可选值：`terminology` / `metaphor` / `cultural_reference` / `cross_domain_term` / `rhetorical_tone` / `structure` / `accuracy` |
| `guidelines` | string[] | 翻译指南 |

**术语 strategy：** `preserve`（保留英文）/ `first_annotate`（首次加注）/ `translate`（直接翻译）

## 翻译阶段（Phase 2）

### 6. paragraph_translate.txt + prompt_builder.py — 段落翻译（核心）

**模板文件：** `src/prompts/longform/translation/paragraph_translate.txt`
**拼装代码：** `src/prompts/prompt_builder.py`

**语言：** 中文 · **模型：** pro · **链路：** A（逐段翻译）/ B（四步法 draft）

**调用点：** `gemini.py` (`_build_translation_prompt`) → `prompt_builder.py` (`build_prompt`)

#### 6.1 翻译配置来源

`prompt_style` 由 `.env` 的 `TRANSLATION_PROMPT_STYLE` 决定（`config.py` → `settings.translation_prompt_style`）：

| .env 值 | builder profile | 注入 `{translation_profile_instructions}` 的内容 |
|---|---|---|
| `original`（默认） | `standard` | 保留密集推理和细腻判断，必要时用更完整的表述 |
| `simplified` | `lean` | 保留所有事实和逻辑，但优先使用更紧凑的中文表达 |

#### 6.2 模板静态骨架

```
角色设定（资深中英双语编辑，硬核科技长文）
翻译配置（{translation_profile_instructions}）
不可违反的原则（准确性 / 重写而非逐译 / 消除翻译腔 / 分析师语气 / 只输出中文）
{dynamic_sections}
原文（{text}）
输出规则（禁 Markdown、格式 token 保护规则）
```

#### 6.3 动态区块详细生成逻辑

`{dynamic_sections}` 由 `_build_dynamic_sections()` 按以下顺序依次拼接（每个区块仅在有数据时才注入）：

##### 6.3.1 隐藏格式 Token（`_build_format_token_section`）
- **触发场景：** 段落含有行内格式（链接、粗体、斜体、行内代码）时。纯文本段落不触发。
- **数据来源：** `paragraph.inline_elements` → `format_token_context()` → `context["format_tokens"]`
- **生成内容：** 5 条 token 操作规则 + 本段 token 列表（最多 6 个，格式：`LINK_1 (link): 原文文本`）

##### 6.3.2 术语表（`_build_glossary_section`）
- **触发场景：**
  - **链路 A：** `select_glossary_terms_for_text()` 从项目+全局合并术语表里只选当前段落命中的条目，使用 `_count_term_occurrences()` 统一做正则边界匹配。
  - **链路 B / C：** `LayeredContextManager._get_relevant_terms()` 从全文分析术语 + prescan + 项目术语覆盖结果中，只保留当前段落命中的条目，并通过 `term_tracker` 修正已使用过的翻译。
- **数据来源：** `context["glossary"]`
- **生成内容：** 最多 30 个术语（`MAX_GLOSSARY_TERMS_IN_PROMPT`），统一格式为 `原文 / 标准写法 / 要求 / 词义`：
  - `preserve` → `原文：HBM；标准写法：HBM；要求：保留英文原文，不加注释`
  - `preserve_annotate`（首次）→ `原文：TSMC；标准写法：台积电；要求：首次出现写”TSMC（台积电）”，后文写”TSMC”`
  - `preserve_annotate`（已出现）→ `原文：TSMC；标准写法：台积电；要求：本段直接写”TSMC”`
  - `first_annotate`（首次）→ `原文：TSMC；标准写法：台积电；要求：首次出现写”台积电（TSMC）”，后文写”台积电”`
  - `first_annotate`（已出现）→ `原文：TSMC；标准写法：台积电；要求：本段直接写”台积电”`
  - 有 `note` → `原文：wafer；标准写法：晶圆；要求：直接使用该写法；词义：这里指芯片制造中的硅片基底`
  - 其他 → `原文：chiplet；标准写法：小芯片；要求：直接使用该写法`

**FIRST_ANNOTATE / PRESERVE_ANNOTATE 动态降级：** `_build_glossary_section()` 接受 `term_usage` 参数。对于 `first_annotate` 和 `preserve_annotate` 策略的术语，如果已在之前的段落中使用过，则不再输出首次标注要求。`term_usage` 来源：
- **链路 B/C：** 由 `TermUsageTracker.used_translations` 提供，断点续传时通过 `_prepopulate_term_tracker()` 预填充。
- **确认流重翻：** 由 `build_term_usage_from_project()` 动态扫描已译段落构建，注入 `TranslationContext.term_usage`。
- 两个函数的透传路径：`build_prompt()` / `build_retranslation_prompt()` → `_build_glossary_section()` → `render_glossary_prompt_block(term_usage=...)`。

##### 6.3.3 已学习的翻译规则（`_build_rules_section`）
- **触发场景：** 全局规则库非空时触发。
- **数据来源：** `memory_service.get_rules_for_prompt()` → `context["learned_rules"]`（`List[str]`）
- **存储位置：** `data/global_memory.md`
- **生成内容：** 最多 20 条 / 600 字符（`MAX_RULES_IN_PROMPT` / `MAX_RULES_CHARS`），自然语言规则原样输出到 `## 已学习的翻译规则`

##### 6.3.4 额外翻译指令 / 修订任务（`_build_instruction_section`）
- **"额外翻译指令"分支**（有 instruction，无旧译文）：首次翻译时用户附带额外指令（少见）。
- **"修订任务"分支**（有旧译文）：用户对已有译文重新翻译。
- **不触发：** 首次全文翻译、四步法 draft（无 instruction 也无旧译文）。
- 仅在 `include_revision_task=True` 时检查。

##### 6.3.5 上下文（`_build_context_section` 第一部分）
- **链路 A：** 通常不触发（不传 `article_title`/`article_theme`）。
- **链路 B / C：** 从全文分析注入，总是触发。
- **内容：** 文章标题、主题、当前章节、标题链、目标读者

##### 6.3.6 结构说明（第二部分）
- 链路 A 不触发；B / C 总是触发。
- 来源：全文分析的 `structure_summary`

##### 6.3.7 章节角色（第三部分）
- 仅 B / C 触发（需要 section role 分析结果）。
- 内容：角色、与上一章节关系、核心论点、翻译注意事项。
- 当前统一限额：核心论点最多 4 条，翻译注意事项最多 4 条。

##### 6.3.8 风格约束（第四部分）
- B / C 总是触发（`style.translation_voice`）；A 不触发。
- 当前统一限额：附加 style notes 最多 4 条。

##### 6.3.9 翻译风险（第五部分）
- 仅 B / C 触发（全文分析 `challenges`）。
- 当前统一限额：高风险挑战最多 4 条。

##### 6.3.10 前文已确认译文（第六部分）
- **链路 A / B：** 当前段落不是 section 第一段时触发。C 不走此区块。
- 取最近 **1 段**，按句子截断（上限 120 字符）。

##### 6.3.11 下文预览（第七部分）
- **链路 A / B：** 当前段落不是 section 最后一段时触发。C 不走此区块。
- 取 **1 段**，按句子截断（上限 160 字符）。

**截断规则（`_truncate_by_sentence`）：** 超限时按句末标点切分，固定保留最后一句，从第一句开始向后累加，中间用 ` …… ` 连接。

#### 6.4 格式 Token 容错与纯链接脱水（运行时）

这部分不是 prompt 模板，而是翻译前后处理逻辑（`src/core/format_tokens.py`）：

1. **纯链接脱水（Pre-LLM）**
   - `is_dehydratable_link_cluster()` 判断段落是否为“仅 link span + 标点/空白分隔”的安全结构。
   - 命中后 `build_dehydrated_link_payload()` 直接生成 `tokenized_text`，跳过 LLM，降低 token 丢失风险。
2. **本地归一化 + 严格校验（Post-LLM）**
   - `build_translation_payload()` 会先调用 `canonicalize_tokenized_markup()`，把近似 wrapper（如全角括号）归一成标准 `[[[TOKEN|...]]]`。
   - 然后调用 `validate_tokenized_text()` 做严格校验（缺失、重复、意外 token、code token 改写、空 token）。
3. **失败自动修复回退（Flash）**
   - 若校验失败且注入了 `token_repairer`，会调用 `llm.repair_format_tokens(...)`（Gemini 默认 `model="flash"`）尝试仅修 token wrapper/ID。
   - 修复结果再次校验；只要 `issues` 数量下降，就采用修复结果。
4. **最终落库策略**
   - 仍有格式问题时：`tokenized_text=None`，`format_issues` 保留详细错误，`text` 继续保存可读译文。
   - 导出阶段仍保持硬校验，block 重建失败会抛 `FormatRecoveryError`。

当前接入范围：
- 链路 A：`TranslationAgent.translate_paragraph()` / `retranslate_paragraph()`
- 链路 B：`FourStepTranslator._translate_single_paragraph()` / `_step_refine()`
- 链路 C：`BatchTranslationService._translate_section_batch()`（纯链接脱水） + `_apply_section_batch_translations()`（校验/修复）
- 图注翻译：`ImageProcessingService._translate_caption()`

#### 确认流快捷重译指令

快捷重译指令以后端 `confirmation_models.py` 中的 `RETRANSLATE_OPTIONS` 为单一事实来源。前端通过 API 拉取选项，点击后以 `option_id` 调用重译接口，由 `resolve_retranslate_instruction()` 解析为实际 instruction：

| 快捷指令 | `option_id` | 说明 |
|---|---|---|
| **可读性** | `readable` | 从句法结构层面重写：控制句长、去被动、删冗余连接词、调整语序、消除连续"的" |
| **更地道** | `idiomatic` | 从表达层面重写：具体代替空泛、简化结构、避免同义词循环和公式化开头 |
| **更专业** | `professional` | 提升专业表达：保留判断力度、产品代号英文、删套话、数据段落拆分重组 |

各选项发送给模型的完整 instruction：

**readable — 可读性（消除翻译腔）**

```
请从句法结构层面重写译文：
1. 优先将长句控制在 30 字以内，超长句必须拆分。
2. 被动语态改为主动语态。
3. 删除不承载逻辑的连接词（如无意义的“此外”“然而”）。
4. 调整为自然中文语序，直接用动词，避免“对……进行……”结构。
5. 连续三个“的”时优先改写句式消除。

示例：
× 此外，该技术对芯片性能的提升效果非常显著。
√ 这项技术大幅提升了芯片性能。
```

**idiomatic — 更地道（自然流畅表达）**

```
请从表达层面重写译文：
1. 用具体事实替代空泛描述。
2. 用简单结构替代复杂绕行表达。
3. 避免同义词堆叠（如“提升/增强/优化”在同一段反复出现）。
4. 避免每段都用“首先/其次/最后”等模板化开头。
5. 非正式语境下，减少“基于/鉴于/旨在”等书面词，改为“根据/考虑到/为了”等自然表达。

示例：
× 基于上述分析，该公司旨在通过技术创新实现业务的全面提升。
√ 根据以上分析，该公司正通过技术创新推动业务增长。
```

**professional — 更专业（科技媒体风格）**

```
请重写译文以提升专业表达：
1. 保留原文观点和判断力度，不要弱化。
2. 产品和技术代号保留英文，行业术语和金融术语优先中文。
3. 删除不承载信息的套话（如“值得注意的是”“众所周知”）。
4. 数据密集段落可适当拆分重组，保持信息密度。
5. 参考术语表中已有译法，保证前后一致。
```

用户也可输入自定义指令。指令定义在 [confirmation_models.py](../src/api/routers/confirmation_models.py) `RETRANSLATE_OPTIONS`，前端面板见 [RetranslatePanel.tsx](../web/frontend/src/features/confirmation/components/panels/RetranslatePanel.tsx)。

# 六、项目导出

系统支持将翻译完成的项目导出为 Markdown 格式，且提供中英文双语选择。

## 6.1 导出格式与逻辑

| 格式 | 参数 (`format`) | 逻辑 | 文件后缀 |
|---|---|---|---|
| **中文译文** | `zh` | 重建项目 block，导出完整的中文译文 Markdown。 | `_zh.md` |
| **英文原文** | `en` | 直接导出项目原始的 `source.md` 文件。 | `_en.md` |

## 6.2 存储路径

所有导出文件均存储在项目目录的 `exports/` 子文件夹下：
`projects/<project_id>/exports/<filename>_<zh/en>.md`

## 6.3 核心实现

- **文件名归一化**：`ProjectManager._sanitize_export_filename_component()` 负责处理非法字符。
- **双语分支**：`ProjectManager.export()` 根据传入的 format 调用 `export_markdown()`（中文）或 `export_source_markdown()`（英文）。
- **前端集成**：在文档页侧边栏的导出组件中，用户可以通过下拉菜单选择语言版本。

## 6.4 Markdown 安全后处理

中文导出在 block 重建和 token 恢复完成后、写入文件前，统一调用 `src/core/markdown_postprocess.py` 的 `postprocess_markdown()` 执行安全后处理：

| 处理 | 说明 |
|---|---|
| 转义裸 `$` | 防止 MathJax/KaTeX 误触发（如 `$100` → `\$100`），保留 `$$` 数学块和已转义 `\$` |
| 转义裸 `<` / 行内 `>` | 防止 Markdown 渲染器误解析为 HTML 标签，保留真正的 HTML 标签和注释 |
| CJK-Latin 空格 | 自动在中文字符与英文字母/数字之间补半角空格 |
| 规范化空行 | 超过 2 个连续空行压缩为 2 个 |
| 保护区域 | 代码块、行内代码、图片、链接、HTML 标签/注释在处理期间被隔离，不被误转义 |

集成入口：`ProjectManager.export_markdown()` / `generate_preview()` / `MarkdownExporter.export_sections()` / `export_bilingual()`
