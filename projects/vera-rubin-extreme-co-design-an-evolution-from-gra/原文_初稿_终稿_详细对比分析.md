# 原文 / 初稿 / 终稿详细对比分析

## 文档目的

这份文档的目标，是为后续修改翻译 Prompt 提炼出真正可执行的经验。这里保留两类分析，而且两类都重要：

1. **术语、专名、措辞层面**：这些问题虽然看起来细，但它们确实会影响技术识别度、行业语感和作者语气，不能删。
2. **句法、语序、拆句、去翻译腔层面**：这是初稿和终稿拉开差距的更深层原因，也是你后续最该反馈进 Prompt 的部分。

换句话说，这份文档不是只讲“词”，也不是只讲“句法”，而是把两者合并起来：先保留原先发现的术语与表达问题，再把句法和段落层面的分析补全。

## 总判断

初稿的主要问题不是“大量误译”，而是**中文技术分析文体没有完全立住**。它大体能传达信息，但经常保留英文句子的骨架，导致出现两层问题：

第一层是显性的：

1. 产品名、平台名、模块名有时过度汉化。
2. 行业术语不稳定，或不够贴近真实中文技术语境。
3. 局部喜欢补强语气，出现“统治力”“回击”“极为深远”这类比原文更重的表达。

第二层是更根本的：

1. 句子主干不清，目的、原因、结果挤在一起。
2. 抽象名词过多，像“存在结构性的利用不足”“部署的权衡逻辑”这类句子明显带翻译腔。
3. 后置定语和括号信息处理不好，句尾过重。
4. 长句虽然信息完整，但层次没有拉开，读起来很累。
5. 段落推进方式偏“逐句搬运”，不像成熟中文分析文那样“先判断、后展开”。

终稿更好的根本原因，不是“词汇更高级”，而是它更愿意为了中文阅读习惯去重建句子、段落和论证顺序。

## 第一部分：术语、专名与措辞层面的问题

这一部分保留原先的分析，但把高度重复的同类问题适度压缩归并。

### 1. 产品名、平台名、芯片代号不宜整体汉化

代表例子：

- `Vera Rubin` -> 初稿 `维拉鲁宾` -> 终稿 `Vera Rubin`
- `Rubin / Blackwell` -> 初稿 `鲁宾 / 布莱克威尔` -> 终稿 `Rubin / Blackwell`

分析：

这类名称在中文半导体语境中，通常直接保留英文。初稿的整体汉化不算误译，但会削弱技术识别度，也会让术语体系不稳定。终稿保留英文，更接近中文产业分析文章的实际写法。

结论：平台名、产品代号、芯片代号默认保留英文，除非中文圈已有非常成熟的固定译名。

### 2. 高频核心概念要术语化，不要每次临场翻

代表例子：

- `cableless` -> 初稿 `无线缆设计` -> 终稿 `无缆化设计`
- `dense` -> 初稿 `密集` -> 终稿 `稠密`
- `fabric` -> 初稿直接保留 `Fabric` -> 终稿 `网络互连`

分析：

初稿的问题不只是某个词不自然，而是高频概念没有先做术语化处理，导致篇内风格漂移。终稿更像是先确定一个可复用的术语，再贯穿全文。

结论：对会反复出现的核心概念，要优先确定“中文主称”，而不是每次边看边翻。

### 3. 行业术语要贴近真实中文语境，而不是只求字面对应

代表例子：

- `transceiver cages` -> 初稿 `光收发器笼子` -> 终稿 `光收发器笼位`
- `manifold` -> 初稿 `分水器` -> 终稿 `歧管`
- `busbar` -> 初稿 `汇流排` -> 终稿 `母排`
- `flyover cable` -> 初稿 `架空线缆` -> 终稿直接保留 `Flyover 线缆`

分析：

初稿很多地方都“看得懂”，但不够像硬件、液冷、电源、PCB 领域真实会说的话。终稿不一定处处都中文化得更多，但它更尊重行业里已经形成的表达习惯。

结论：技术翻译不是词典翻译。优先考虑中文圈真实使用习惯，再考虑字面完整度。

### 4. 来源标题、模块全称、辅助信息的位置，可以比正文更中文化

代表例子：

- 来源标题 `VR NVL72 Component BoM and Power Budget Model`
- 图注中的 `Strata Module / Bianca Module`
- `Trusted Platform Module / Datacenter Secure Control Module`

分析：

正文主体里保留产品英文名没有问题，但图注、来源、模块全称、辅助标题如果保留过多英文，会让中文文面过硬，甚至出现中英混搭感。这个问题我已经在终稿文件里直接帮你改过一轮。

结论：正文主体和辅助信息可以采用不同保留策略。正文偏专业英文没问题，辅助信息则应优先保证中文扫读效率。

### 5. 初稿容易把作者语气再往上推半级

代表例子：

- `supremacy` -> 初稿 `统治力` -> 终稿 `优势`
- `responds` 语境 -> 初稿 `予以回击`
- `will determine who are winners or losers` -> 初稿转成 `市场洗牌效应将极为深远`

分析：

初稿常见的问题不是方向错，而是把中性分析写得更像宣传判断。终稿更稳，因为它会控制评价强度，不轻易替作者拔高情绪。

结论：遇到带评价色彩的英文词，不要自动选中文里最重的词；先判断作者是在“描述优势”，还是在“下结论”。

### 6. 商业、金融、研究分析类表达要像中文研究报告

代表例子：

- `higher Capital Cost of Ownership` -> 初稿 `更高的资本总拥有成本` -> 终稿 `资本成本口径下的总拥有成本`
- `what Nvidia plans to do with Groq` -> 初稿 `针对 Groq 的应对策略` -> 终稿 `对 Groq 的计划`
- `memory economics` -> 初稿 `内存经济效益` -> 终稿 `内存经济性`

分析：

这一类不是技术术语问题，而是研究文体问题。初稿往往字面忠实，但中文不够像产业研究或金融研究文章。终稿则更贴近中文研究写作习惯。

结论：TCO、资本开支、商业假设、并购/授权/合作等内容，不能按技术说明书的翻法处理。

## 第二部分：句法、语序、拆句与去翻译腔

这一部分是你更关心的重点，也是决定终稿“更像自然中文”的核心。

## 一、句子主干：终稿会先把句子立住

### 1. 目的状语后移，因果链更顺

原文：`Rack scale integration and assembly have become more challenging... VR NVL72 has a much more holistic design with a modular ... to improve integration efficiency and throughput.`

初稿：`随着每个组件都被推向极限，同时还要兼顾成本效率的优化，机架级的集成与组装正变得愈发困难。为了提升集成效率与吞吐量，与Grace Blackwell相比，VR NVL72采用了模块化方案，其整体设计更具全局性。`

终稿：`随着每个组件都被推向极限，同时还要兼顾成本效率，机架级集成与装配正变得越来越困难。相比 Grace Blackwell，VR NVL72 采用了模块化方案，其整体设计更具全局性，目标就是提升集成效率与吞吐。`

分析：

初稿把“为了提升集成效率与吞吐量”提得太早，导致句子重心摇摆。读者读到中间会暂时分不清：这句到底在写“为什么更难”，还是在写“为什么采用模块化”。

终稿把结构拉直成：

1. 问题：装配更难；
2. 方案：采用模块化；
3. 目标：提升效率与吞吐。

这就是很典型的中文优化：**目的状语后移，先把主干说清。**

### 2. 导览句从“平移英文结构”改成“中文提纲句”

原文：`In the sections below, we will discuss... Then, we will discuss...`

初稿：`在下文中，我们将深入芯片层面，剖析维拉鲁宾平台的6款芯片产品。随后，我们将从设计视角探讨机架与计算托盘从Grace Blackwell向维拉鲁宾的演进，并分析其对线缆、连接器、PCB、散热、机械结构与供电等组件的影响。`

终稿：`在下面的章节中，我们将先深入芯片层面，剖析 Vera Rubin 平台的这 6 款芯片产品。随后，我们会从设计视角探讨计算托盘与机架如何从 Grace Blackwell 演进到 Vera Rubin，以及这种演进对线缆、连接器、PCB、散热、机械结构和供电所带来的影响。`

分析：

初稿基本是把英文提纲结构平移过来：

- `在下文中，我们将……`
- `随后，我们将……`
- `并分析其对……的影响`

句法完整，但很像翻译。

终稿更自然，主要因为它做了三件事：

1. 加了 `先`，让顺序更像中文展开。
2. 把 `从……向……的演进` 改成 `如何从……演进到……`，让动词真正带动句子。
3. 把 `其对……的影响` 改成 `这种演进对……所带来的影响`，指代更清楚。

结论：终稿更会把英文提纲句改写成中文导览句。

### 3. 总述句先下判断，再解释

原文：`Rubin’s design is a logical evolution from Blackwell...`

初稿：`鲁宾的设计是布莱克威尔合乎逻辑的演进，它转向了3纳米工艺，并将I/O拆分为芯粒，同时保留了两颗光罩极限尺寸（reticle-sized）裸片搭配8栈HBM的相同基本结构。`

终稿：`Rubin 的设计是 Blackwell 合乎逻辑的演进：制程推进到 3nm，同时将 I/O 拆分为芯粒（chiplet），但仍保留了两颗光罩尺寸裸片（reticle-sized die）配合 8 组 HBM 堆栈的基本结构。`

分析：

初稿句子里的信息都对，但主次不分。终稿用冒号把它改成：

- 前半句：总判断；
- 后半句：为什么这么判断。

这很像中文分析文常见的句法，而不是英文直译。

## 二、抽象名词：终稿更少“名词腔”，更多“判断句”

### 4. 从“抽象名词堆叠”改成“动作或判断”

原文：`changes the deployment calculus`

初稿：`独立/专用机架的方向实质性地改变了部署的权衡逻辑。`

终稿：`转向独立/专用机架，会实质性地改变部署逻辑。`

分析：

初稿的问题很典型：

- `……的方向`
- `……的权衡逻辑`

全是抽象名词，读起来像翻译稿。

终稿把它们动词化了：

- `转向……`
- `改变部署逻辑`

这就是自然中文的关键：**少用抽象名词壳，多让动词承载句子。**

### 5. 从“存在结构性的利用不足”改成自然判断句

原文：`is structurally underutilized`

初稿：`HBM 增加的带宽在 Prefill 环节中存在结构性的利用不足。`

终稿：`HBM 的额外带宽在预填充场景中，从结构上就难以被充分利用；`

分析：

初稿完全没错，但翻译腔很重。问题不在术语，而在句法：

- `存在……利用不足`

这种写法很像学术直译，不像正常中文分析文。

终稿改成：

- `从结构上就难以……`

一下就变成了中文里自然的判断句。

### 6. 从空泛评论改成具体判断

原文：`will determine who are winners or losers`

初稿：`其产生的下游影响与市场洗牌效应将极为深远。`

终稿：`将决定谁会在这场规模高达 5000 亿美元的 Rubin 基础设施扩建浪潮中成为赢家，谁会成为输家。`

分析：

初稿不是误译，而是把原文一个具体判断，翻成了一个抽象评论。结果中文虽然“正式”，但更空。

终稿回到原文的直接判断结构：谁赢、谁输。反而更像成熟中文分析写法。

结论：终稿更会把英文里的结论句保持为**具体判断**，而不是抽象概念。

## 三、长句重组：终稿更会拆层，而不是一口气压平

### 7. 技术长句按“机制 -> 数字 -> 对照”展开

原文：`The move to HBM4 means... 10.8 GT/s... 22TB/s... at the same 288GB capacity...`

初稿：`在内存方面，升级至 HBM4 意味着每个堆叠的总线宽度翻倍，运行速率达到 10.8 GT/s，从而实现 22TB/s 的总带宽；在与 GB300 保持相同 288GB 容量的情况下，其带宽达到了布莱克威尔的 2.75 倍。`

终稿：`在内存方面，转向 HBM4 意味着每个堆栈的总线宽度翻倍，在 10.8GT/s 下可实现总计 22TB/s 的带宽；在容量仍维持 GB300 同样 288GB 的前提下，带宽达到了 Blackwell 的 2.75 倍。`

分析：

初稿的问题不是太长，而是句子内部没有层次，读起来像参数说明书。

终稿把信息分成三层：

1. 机制：转向 HBM4；
2. 数字：10.8GT/s、22TB/s；
3. 对照：容量不变、带宽上升。

这会让技术句更像分析，而不是照抄规格。

### 8. 公式段不是只“翻公式”，而是先把读者带进去

原文：`By one 1.6T logical port, we mean...`

初稿：`所谓“一个 1.6T 逻辑端口”，是指……`

终稿：`这里说的一个 1.6T 逻辑端口，指的是……`

分析：

这个例子看上去差别很小，但很能说明问题。

初稿是“术语定义句”；终稿是“解释句”。终稿多了一个很中文的引导动作：`这里说的……`

公式段最怕的是读者还没进场，就先被定义砸到。终稿会先把语气放低，把读者领进去。

### 9. 长段中的多层信息，终稿会先总后分

代表段落：散热供应链影响、TCO 分析、ICMS / CMX。

初稿的普遍问题是：信息都在，但一股脑往前推。比如某一段里同时讲趋势、设备变化、供应商名单、二级影响，结果句子每部分都对，但主结论不突出。

终稿更常见的组织方式是：

1. 先给结论；
2. 再展开原因；
3. 最后补细项或例子。

这并不是删信息，而是给信息分工。对中文长段来说，这一点比“句子表面更短”更重要。

## 四、括号和补充说明：终稿更会把信息融进主句

### 10. “通过采用……（解释）”是典型翻译腔

原文：`with an updated 3rd generation Transformer Engine that replaces...`

初稿：`通过采用更新的第三代Transformer引擎（取代了前几代架构中的2:4结构化稀疏）……`

终稿：`借助升级后的第三代 Transformer Engine——它用新的机制取代了前几代中的 2:4 结构化稀疏——……`

分析：

初稿的问题在于：

1. `通过采用……` 本身就偏硬；
2. 括号让句子像在插批注。

终稿的处理更像中文作者在解释：

- 先用 `借助` 降低硬度；
- 再用破折号把解释嵌进主句。

这样读起来不是“正文 + 注释”，而是一个完整句子。

### 11. 译者式校注会破坏文气

代表例子：Strata 段中初稿写 `（原文为 1,534GByte）`。

问题不只是这句多余，而是它把整段视角从“介绍系统结构”拉回了“展示译者正在核数字”。中文分析文一旦频繁出现这种译者动作，读者就会不断意识到自己在看译文。

终稿删掉这类内容后，段落的叙述视角更统一。

结论：如果用户不是要校勘稿，正文里就不该频繁出现译者式补注。

## 五、语序调整：终稿更符合中文的信息出场顺序

### 12. 后置定语不能整串拖到句尾

原文：`the management modules are the only other two components within the compute tray that Nvidia allows customization for`

初稿：`除 BlueField-4 外，供电模块和管理模块是算力托盘中英伟达允许定制的仅有的另外两个组件。`

终稿：`除 BlueField-4 外，供电模块和管理模块，是英伟达在计算托盘内部仅有的另外两个允许客户定制的部件。`

分析：

初稿把修饰信息都堆到了后面，句尾非常重。终稿虽然也长，但把限制条件分散到了句中不同位置，让句子边读边清楚。

这类问题在初稿里非常常见，本质上是**英文后置定语没有经过中文重排**。

### 13. 把定语结构改写成情景结构

原文：`especially for longer sequence length queries that are becoming popular between turns or tool calls`

初稿：`特别是对于轮次间或工具调用间日益流行的长序列长度查询。`

终稿：`尤其是在较长序列长度查询越来越常见、轮次之间或工具调用之间都需要保留状态的情况下。`

分析：

初稿的问题在于它还是一个长定语，句法像英文。

终稿把它改成了中文更自然的“情景展开”：

- 查询越来越常见；
- 轮次之间需要保留状态。

这比“对于……的……查询”自然很多。

## 六、段落推进：终稿更像中文分析文，而不是逐句翻译

### 14. 初稿常常“逐句忠实”，终稿更重视段落节奏

以导语、ICMS 段、散热段为例，初稿通常是一句接一句按原文顺序翻下来，句子之间缺少明显的中文层次信号。终稿则更愿意做这些动作：

1. 先给读者一个结论或方向；
2. 再补背景；
3. 再落到技术机制；
4. 最后补数字、例子或供应链后果。

所以终稿读起来会像“文章”，初稿读起来更像“句子序列”。

### 15. 终稿更会使用中文里的组织句

例如这些表达：

- `目标就是……`
- `这里说的……`
- `从工程上讲……`
- `归根结底……`
- `另一条路径当然是……`

这些都不是华丽修辞，而是中文分析文里非常重要的组织器。它们能帮助读者理解“这句话在段里起什么作用”。

初稿缺的恰恰就是这一层，所以它经常显得“每句都对，但连起来不顺”。

## 第三部分：更深一层的归纳

前两部分已经把问题类型拆开了。为了让这份文档更适合拿去改 Prompt，这里再往上一层，总结三件更重要的事：

1. 怎么判断一句话到底有没有翻译腔；
2. 为什么这些问题会反复同时出现；
3. 哪些问题优先级最高，最值得先修。

### 一、如何判断一句话是否有翻译腔

很多时候，一句中文并没有错，但你读完会觉得“像译文”。这种感觉其实可以拆成几个很具体的判断标准：

#### 1. 看句子是不是“名词太多、动词太弱”

如果一句话主要靠这些结构撑起来：

- `……的逻辑`
- `……的方向`
- `……的影响`
- `存在……`
- `进行……`
- `实现……的提升`

那它大概率就已经开始往翻译腔走了。

原因很简单：英文很擅长用抽象名词组织逻辑，中文更擅长用动作、判断和顺序组织逻辑。终稿明显更会把抽象名词还原成动作句或判断句。

#### 2. 看句子是不是“主干出来得太晚”

如果读者要读到后半句，甚至读完整句，才知道句子真正想说什么，那么这句大概率也不够自然。

初稿里常见的问题是：

- 前面先堆条件、背景、修饰；
- 后面才出现真正的主干；
- 句尾还压着很重的限定成分。

终稿的一个核心改进，就是更愿意让主干先出来，让读者尽早知道句子“在说什么”。

#### 3. 看句子是不是“像注释，不像叙述”

如果一句中文大量依赖括号、英文原词、术语补注来维持精度，读者就会不断被打断。

这不一定代表术语错了，而是说明信息的组织方式仍然偏“注释式”。终稿更成熟的地方，在于它常常把原本放进括号里的信息消化进主句，使句子仍然是连续叙述，而不是正文外接注解。

#### 4. 看句子是不是“像在对应英文结构”

比如这些表达，往往都说明句子还没有完全转成中文：

- `在下文中，我们将……`
- `特别是对于……的……`
- `通过采用……其……可……`
- `存在结构性的……`
- `独立/专用机架的方向……`

这些句子不是不能用，而是它们往往保留了英文原结构的影子。终稿更像是“先忘掉英文长什么样，再重新写成中文”。

#### 5. 看句子是不是“意思对，但发力不对”

翻译腔不只是句法问题，也包括语气问题。初稿有一类问题很隐蔽：词义方向没错，但中文发力点偏了。

典型表现是：

- 原文只是判断，译文变成评论；
- 原文只是优势，译文变成压倒性胜利；
- 原文只是说明后果，译文却加了“极为深远”这类扩大语气。

所以，判断翻译腔不能只看“句子顺不顺”，还要看中文是不是在替作者额外说话。

### 二、为什么这些问题总是一起出现

初稿里的问题看起来分散：有的是术语，有的是语序，有的是语气，有的是长句。但它们其实往往来自同一个更底层的习惯：

**模型默认在做“逐句映射”，而不是“按中文写作习惯重写”。**

一旦采用“逐句映射”模式，就会自然出现下面这些连锁反应：

1. 更倾向保留英文原有句法关系。
2. 更不敢调整语序和段落顺序。
3. 更依赖括号来保住精度。
4. 更容易保留抽象名词结构。
5. 更容易把作者原本较克制的判断翻成中文里的重语气。

这也是为什么术语问题和句法问题不能完全拆开看。很多时候，术语不稳，不是因为不知道术语，而是因为句子还处在“映射英文”的思维里；而句子之所以不自然，也常常是因为术语处理方式把句法拖硬了。

### 三、最值得优先修的不是“所有问题”，而是几个杠杆点

如果你后续真的要拿这份文档去改 Prompt，不需要一次把所有问题全修完。更高效的做法，是先抓最有杠杆的几个点。

我建议优先级如下：

#### 第一优先级：允许主动重组句法

这是最关键的一条。只要模型还默认自己不能重写句法，那它就很容易继续保留英文结构，进而连带生出抽象名词堆叠、句尾过重、括号过多等问题。

#### 第二优先级：强制减少抽象名词腔

如果 Prompt 明确限制少用这些句型：

- `存在……`
- `进行……`
- `实现……的提升`
- `……的逻辑/方向/结构`

整体中文会立刻自然很多。

#### 第三优先级：要求“先判断、后展开”

无论是单句还是段落，终稿最稳定的优势之一，就是会先给读者一个判断或方向，再展开说明。这个习惯一旦建立，长句和长段都会明显更顺。

#### 第四优先级：控制作者语气，不替原文加码

这条虽然排在第四，但也很重要。因为一旦语气失真，整篇文章就会从研究分析滑向宣传稿，即使句子本身已经很顺，也仍然会显得不成熟。

## 第四部分：把这些分析变成可执行的检查标准

如果你后面要对 AI 初稿做人工验收，可以用下面这套问题直接检查，而不是靠模糊感觉。

### 一、句子层面检查

每读到一个长句，都可以问自己：

1. 这句话的主干是不是在前半句就已经清楚了？
2. 这句话是不是用了太多“……的……”“存在……”“实现……”这类抽象结构？
3. 这句话有没有把英文后置定语整串拖到句尾？
4. 这句话里的括号信息有没有可能并入主句？
5. 这句话是不是在模仿英文结构，而不是自然中文？

如果其中有两条以上回答“是”，这句基本就值得重写。

### 二、段落层面检查

每读到一个长段，都可以问自己：

1. 这段开头有没有先给我一个方向或结论？
2. 这段中间的信息有没有主次，还是全都一个重量？
3. 这段是在展开论证，还是只是在逐句传递信息？
4. 我读完之后记住的是一个判断，还是只记住了一串细节？

如果读完之后只记住细节，没记住判断，通常就说明段落结构还不够中文化。

### 三、语气层面检查

还可以额外问：

1. 这句话是不是比原文更重、更像宣传？
2. 原文的不确定性有没有被保住？
3. 原文的判断有没有被我翻得更空、更虚？

这组检查能防止另一种常见问题：句子看起来顺了，但作者原本的分寸感丢了。

## 第五部分：更适合用于 Prompt 的高层规则

在前面的基础上，我建议把 Prompt 里的要求分成三层，而不是堆一串零散指令。

### 第一层：文体目标

不要只说“翻译自然”，而要明确：

**把英文原文改写成中文半导体研究文章，而不是逐句对应的译文。**

这句话比单纯说“避免翻译腔”更有约束力，因为它直接规定了目标文体。

### 第二层：句法原则

可以明确写：

1. 允许主动调整语序和拆句，只要不改变信息与逻辑关系。
2. 长句优先重建主干，再安排目的、条件、补充说明。
3. 优先使用中文判断句和动作句，减少抽象名词腔。
4. 后置定语不得整串拖到句尾，必要时拆成两句。
5. 括号只保留必要术语补充，能并入主句则并入主句。

### 第三层：文风限制

再加一层限制，防止生成另一种偏差：

1. 不要擅自增强作者语气。
2. 不要添加译者式补注。
3. 不要把具体判断改写成空泛评论。
4. 不要为了显得专业而过度保留英文残片。

这样 Prompt 才会同时兼顾“自然”和“克制”。

## 第三部分：可直接反馈给 Prompt 的经验

如果你的目标是让 AI 初稿更接近终稿，我建议 Prompt 不要只写泛泛的“语言自然、避免翻译腔”，而要写成下面这种具体规则：

1. 产品名、平台名、芯片代号默认保留英文，不要整体音译成中文。
2. 高频核心概念先确定统一术语，再贯穿全文。
3. 允许主动重组语序，只要不改变原文信息和逻辑关系。
4. 翻长句时，先找主干，再决定目的、条件、补充说明放在哪里。
5. 少用“存在……”“进行……”“实现……的提升”“……的权衡逻辑”这类抽象名词腔。
6. 英文后置定语不要整串拖到句尾，必要时拆句。
7. 能写成判断句，就不要写成抽象名词句。
8. 技术解释段要像“作者在解释”，而不是“译者在注释”。
9. 公式段、TCO 段、供应链段要先给读者进入路径，再给细节。
10. 商业、金融、TCO、合作/授权类内容要按中文研究报告文体表达。
11. 正文主体与图注/来源/辅助标题可以采用不同的中英保留策略。
12. 最终输出风格应接近“中文半导体研究文章”，而不是“忠实的逐句直译稿”。

你也可以把这些规则压缩成一句更高层的要求：

**允许为中文可读性主动重写句法与段落顺序，但必须保持原文的信息、论证关系与语气分寸。**

## 最后总结

真正拉开初稿和终稿差距的，不是几个词有没有翻准，而是终稿做到了三件事：

1. 先把中文句子的主干立住；
2. 再把英文里的信息按中文顺序重新排；
3. 最后让整段读起来像中文分析，而不是翻译结果。

术语问题当然仍然重要，所以我保留了前面的分析；但如果你只能优先修一类 Prompt 规则，那最该修的还是句法层面：

**让模型默认可以重写句法，只要忠实于原文信息和论证关系。**

如果再往前走一步，这份文档最终想说明的其实是：

好译文不是“每个词都对上”，而是**读起来不像翻译**。而要做到这一点，术语准确只是底线，真正决定质量上限的，永远是句法、段落和语气控制。
